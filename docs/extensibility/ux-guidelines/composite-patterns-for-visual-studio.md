##  <a name="BKMK_DataVisualization"></a> 데이터 시각화  
  
### <a name="overview"></a>개요  
 차트는 시각적으로 집계 하 고 의사 결정을 개선 하기 위해 데이터를 시각화 합니다. 많은 주의 기울이게 기능 및 작업을 할 수 어떤 참조 거의 의미 하지만 데이터를 처리 하는 사용자가 도움이 수 있습니다.  
  
 사용자는 다음 조건 중 하나가 충족 되는 경우 차트에서 좋습니다.  
  
-   차트를 작업할 수 있는 작업을 식별 하는 사용자가 도움이 될?  
  
-   차트 하면 사용자가 잠재적 변경 사항의 결과 예측할 수 있습니까?  
  
-   차트에 추세를 파악 하 고 패턴을 식별 하는 사용자가 도움이 될?  
  
-   차트 하면 사용자가 더 나은 결정을 내릴 수 있습니까?  
  
-   사용자가 지정된 된 컨텍스트에 있을 수 있으며 특정 질문에 대답 차트 도움이 됩니다.  
  
#### <a name="general-rules-for-charts"></a>차트에 대 한 일반 규칙  
  
-   레이블 데이터 명확 하 게 합니다. 설명 하지 않고 그림은 방금 그림 정말 합니다.  
  
-   축 기울이기 비율을 방지 하기 위해 0부터 시작 합니다. 줄 길이 막대 크기는 데이터 요소 간의 관계를 이해 하는 데 중요 한 시각 신호입니다.  
  
-   하지 인포 그래픽 차트를 만듭니다. 인포 그래픽은 데이터에 대 한 꾸밈 표시 되며 기본 만드는 목적은 visual 스토리텔링 있습니다. 수 (및 해야) 차트 시각적으로 매력적인 있지만 자체적 데이터 수 있도록 합니다.  
  
-   Skeumorphism, 그림 막대 그래프, 대비 hashmarks 및 기타 인포 그래픽 터치 하지 마십시오.  
  
-   장식 요소와 3D 효과 사용 하지 마십시오. 경우에 사용 정보를 이해 하는 사용자의 기능에 필수적인 진정으로 있습니다.  
  
-   두 개 이상의 색 읽고 올바르게 해석 하기 어렵게이 종류의 차트를 만들 수 있습니다 때 여러 선 및 채우기, 사용 하지 마십시오.  
  
-   개념을 이해 하거나 데이터와 상호 작용 하는 유일한 방법은으로 차트 (또는 원하는 그림)을 사용 하지 마십시오. 이 시각 장애가 있는 사용자에 대 한 문제가 발생합니다.  
  
-   페이지에서 무료 또는 장식 요소와 차트를 사용 하지 마십시오. 즉, 문제를 해결 하는 모든 값 또는 도움말 사용자가 차트를 추가 하지 않는, 경우 사용 하지 않아도 됩니다.  
  
### <a name="chart-types"></a>차트 종류  
 Visual Studio에서 사용 되는 차트에 가로 막대형 차트, 꺾은선형 차트, 수정 된 원형 차트를 링 차트 또는 "도넛형 차트"와 타임 라인 라고 산 점도 ("차트 클러스터"), 및 Gantt 차트입니다. 각 차트 종류에는 다른 유형의 정보가 통신 하는 데 유용 합니다.  
  
### <a name="other-charting-considerations"></a>다른 차트 고려 사항

##  <a name="BKMK_OnObjectUI"></a> 개체에 UI 및 보기  
 이 섹션 관찰, 코드 peek 보기 라고도 형식의 Visual Studio에 고유한 개체에 UI에 컨텍스트를 제공 합니다.  
  
### <a name="overview"></a>개요  
  
-   개체에 UI 사용자가 부여 자세한 정보 또는 상호 작용의 주 작업에서 떨어뜨리지 않고 합니다.  
  
-   Visual Studio에서 개체에 UI에 대 한 기본 패턴 "관심 지점에서 정보입니다." 라고 합니다.  
  
-   Visual Studio에서 UI 개체에는 인라인 또는 부동 및 중 하나 내구성이 또는 일시적이 지입니다.  
  
    -   코드 미리 보기 뷰는 형식의 Visual Studio에서 UI 개체에 인라인 및 지속적인 됩니다.  
  
    -   부동 및 일시적인 CodeLens는 형식의 Visual Studio에서 UI 개체에는  
  
 코드의 작동 방식을 이해 하거나 해당 코드에 대 한 세부 정보를 찾으려는 컨텍스트를 전환 하 고 다른 콘텐츠 또는 다른 위치로 이동 하는 개발자를 위해 경우가 종종 창. 주 창 떠나면 자신의 원래 작업에 포커스를 손실 될 수 있으므로 이러한 컨텍스트 shifts 파급 효과 수 있습니다. 또한 컨텍스트 원래의 인해 발생 한 다른 UI에 의해 가려질 수는 원래 코드 창 전환 하는 경우에 특히 어려울 수를 가져옵니다.  
  
 개체에 UI 뒤에 오는 "관심 지점에서 정보입니다." 이라는 패턴 이러한 메시지, 팝업 창 및 대화 상자 사용자가 주 작업에 초점을 손실 하지 않고 자세한 설명 이나 대화형 작업을 추가 하는 관련 추가 정보를 제공 합니다. 개체에 UI의 예로 알림 영역에서 아이콘, 맞춤법이 틀린된 단어 및 Visual Studio 2013에 도입 된 미리 보기 뷰 아래의 빨간색 물결 기호 위로 마우스 포인터를 가리킬 때 표시 되는 팝업 창이 있습니다.  
  
### <a name="decision-points"></a>의사 결정 요점  
 Visual Studio 내에서 관심 지점에서 정보의이 패턴을 사용 하려면 여러 가지가 있습니다. 오른쪽 메커니즘 선택 하 고 일관 되 고 예측 가능한 방식으로 구현 전반적인 환경을에 필수적입니다. 그렇지 않으면 사용자가 콘텐츠 자체에서 포커스에 악영향을 하는 일관 되지 않은 또는 혼란 스러운 experience와 함께 표시 될 수 있습니다.  
  
#### <a name="relationships-between-master-and-detail-content"></a>마스터 / 세부 콘텐츠 간 관계  
 관심 지점에서 정보는 관계를 표시 하는 데 사용 됩니다 ("master" 콘텐츠)에 초점을 맞추고 및 추가 사용자가 콘텐츠 간의 콘텐츠 ("detail" 콘텐츠) 관련 합니다. 이 패턴에서는 세부 콘텐츠는 콘텐츠를 사용자가 사용 되 고 마스터 콘텐츠 가깝게 표시 될 수 명확 하 게 관련 되어 있습니다. 추가 정보 또는 마스터 콘텐츠 과도 하 게 많아지지 없이 요약할 수 없는 정보 도구 창 다른 패턴을 따라야 합니다.  
  
-   **항상** 마스터 콘텐츠에 근접의 세부 내용을 표시 합니다.  
  
-   **항상** 세부 콘텐츠는 사용자가 마스터 콘텐츠에 초점을 맞추고 여전히 수 있음을 확인 합니다. 대개는 가장 좋은 방법은이 작업을 수행할 마스터 콘텐츠 최대한 가깝게로 세부 내용을 렌더링 하는 합니다. 마스터 콘텐츠 옆의 팝업 창에서 세부 정보 콘텐츠를 렌더링 하 여 마스터 콘텐츠 아래 세부 콘텐츠 인라인을 렌더링 하 여 수행할 수 있습니다.  
  
-   **하지** 주의 마스터 콘텐츠에서 사용자를 사용 하는 지점에서 정보를 사용 합니다. 사용자가 세부 콘텐츠를 별도로 검토 해야 하는 경우이 작업을 수행할 수 있는 명시적인 작업을 노출 합니다.  
  
#### <a name="design-details"></a>디자인 세부 정보  
 개체에 UI는 것이 좋습니다 결정 하면 4 개의 주요 디자인 고려 사항 가지가 있습니다.  
  
1.  **지 속성:** 콘텐츠 이어야 하는데 지속적인 지 또는 일시적인?   
    사용자가 됩니다 계속 정보를 참조 하거나 상호 작용을 표시 하 시겠습니까? 또는 사용자 정보 보기 신속 하 게 한 다음 주 작업 계속?  
  
2.  **콘텐츠 형식:** 콘텐츠 됩니다 정보 제공 용 이므로, 작동, 또는 탐색?   
    사용자는 마스터 내용에 대 한 추가 세부 정보에 필요 합니까? 마스터 콘텐츠에 영향을 주는 작업을 완료 하는 사용자가 필요 합니까? 또는을 다른 리소스에 사용자 필요 합니까?  
  
3.  **표시기 유형:** 앰비언트 표시기 의미가 있습니까?   
    정보에는 유용한 방법은 요약 및 표시 될 수 과도 하 게 많아지지 마스터 콘텐츠 없이?  
  
4.  **제스처:** 제스처 어떤 사용될지를 호출 하 고 UI를 닫습니다.   
    어떻게 사용자 세부 정보 콘텐츠를 표시 하 고는 다른 페이지로 보낼? 임시 및 지속적인 상태 간에 전환 하려면 고정 같은 제스처를 추가에 있는 값이 있습니까?  
  
 개체에 UI의 주요 구성 요소에 영향을 미치게 됩니다 각 4 개의 의사 결정 사항입니다.

3.  제스처  
  
    -   호출

4.  지 속성 및 커밋 모델

    -   스마트 태그 아이콘  
  
    -   앰비언트 다른 표시기  
  
#### <a name="container-content-presenter-type"></a>컨테이너 (콘텐츠 발표자) 형식  
 관심 지점에서 콘텐츠를 제공 하는 사용할 수 있는 두 가지 주요 옵션에는  
  
1.  **인라인:** 는 인라인 발표자 Visual Studio 2013 코드 편집기에 도입 된 peek 보기와 같은 기존 콘텐츠를 이동 하 여 새 내용에 대 한 공간을 만듭니다.  
  
    -   **원하는** 제공할 사용자가 사용할를 참조 하거나 콘텐츠와 상호 작용 하는 시간이 많이 소요 될 경우 인라인 발표자입니다.  
  
    -   **방지** 있는 다음 주 작업 중단을 최소화 하면서 계속 정보를 한 눈에 보기를 사용자가 증명 하는 경우 인라인 발표자 것이 좋습니다.  
  
2.  **부동:** 부동 발표자는 선택된 된 콘텐츠를 최대한 가깝게 있지만 기존 내용의 레이아웃을 변경 하지 않습니다. 다양 한 전략 채택할 수를 통해 콘텐츠 고정 해제 패널을 표시 하는 등의 선택한 기호에 공백을 사용할 수 있는 가장 가까운 합니다.  
  
    -   **원하는** 있는 다음 주 작업 중단을 최소화 하면서 계속 정보를 한 눈에 보기 하려고 것 사용자가 증명 하는 경우 발표자 부동 합니다.  
  
    -   **방지** 를 참조 하거나 콘텐츠와 상호 작용 하는 시간이 많이 지출 하 합니다 사용자가 증명 하는 경우 발표자 부동 있는 있습니다.  
  
#### <a name="content-type"></a>콘텐츠 형식  
 세 가지 주요 형식의 모든 개체에 UI 컨테이너 내에 표시 될 수 있는 콘텐츠 있습니다. 어떠한 조합의 이러한 종류의 정보를 표시할 수 있습니다. 세 가지 유형은 다음과 같습니다.  
  
1.  **알림:** 대부분 개체에 UI 컨테이너 몇 가지 종류의 정보 콘텐츠를 표시 합니다. 콘텐츠는 환경의 현재 상태에 대 한 정보를 나타낼 수 있습니다 또는 환경의 잠재적인 이후 상태에 대 한 정보를 나타낼 수 있습니다. 예를 들어 기존 코드를 리팩터링 하는 등 특정 명령의 효과를 표시 사용할 수 없습니다.  
  
    -   **항상** 표시할 정보의 정식 표현을 사용 합니다. 예를 들어, 코드 완성 구문 강조 표시 된 코드와 같습니다 및 글꼴 및 사용자가 설정한 다른 환경 설정을 적용 해야 합니다.  
  
    -   **항상** 가능한 마스터 내용으로 동일한 정보에 표시 되는 정보 제공 용 이므로 콘텐츠 위에 있는 동작을 지 원하는 것이 좋습니다. 예를 들어 개체에 UI 컨테이너 내에서 기존 코드를 제시 고려해 검색 하 고 해당 코드를 수정할 수 있는 기능을 지원 합니다.  
  
    -   **항상** 잠재적인 이후 상태 표시 합니다. 콘텐츠 정보를 제공 하는 경우 다른 배경색을 사용 하는 것이 좋습니다.  
  
2.  : 실행 가능한 일부 개체에 UI 컨테이너 마스터 내용을 리팩터링 작업을 수행 하는 등 일부 작업을 수행 하는 기능을 제공 합니다.  
  
    -   **항상** 정보 콘텐츠 별도로 실행 가능한 명령을 위치를 지정 합니다.  
  
    -   **항상** 사용 하도록 설정 하 고 적절 한 경우 작업을 사용 하지 않도록 설정 합니다.  
  
    -   **항상** 대화 상자 안의 명령을 나타내는 표준의 지침을 참조 하십시오.  
  
    -   **항상** 최소 절대 수 있는 개체에 UI 컨테이너에 노출 되는 작업의 수를 유지 합니다. 개체에 UI와의 상호 작용 간단 하 고 빠른 환경 이어야 합니다. 사용자를 사용할 개체에 UI 컨테이너 자체 관리에 대 한 에너지 없어야 합니다.  
  
    -   **항상** 개체에 UI 컨테이너 종료 되거나 해제 될 시기와 방법을 고려해 야 합니다. 모범 사례로, 마스터 및 세부 콘텐츠 간의 대화를 종료 하는 모든 작업 또한 닫아야 개체에 UI 컨테이너 해당 작업을 호출할 때.  
  
3.  **탐색:** 일부 개체에 UI 컨테이너 창 또는 사용자의 웹 브라우저에서 MSDN 문서를 여는 등의 응용 프로그램을 다른 사용자를 사용 하는 링크를 포함 합니다.  
  
    -   **항상** "열기" 탐색 링크를 다른 콘텐츠를 탐색 하 여 사용자가 달라진 것은 앞에 추가 합니다.  
  
    -   **항상** 조치 가능한 링크에서 이동 하는 링크를 구분 합니다.  
  
#### <a name="ambient-indicators-optional"></a>앰비언트 표시기 (옵션)  
 앰비언트 표시기는 코드의 나머지 부분에서 대비 되는 색으로 표시 되는 텍스트를 포함 하 여 감지 하기 어려운 또는 tickler 물결선 밑줄, 스마트 태그 아이콘 등의 기호를 포함 하 여 확실 한 가능 합니다. 앰비언트 표시기 관련 추가 정보가의 가용성을 통신합니다. 이상적으로 상호 작용에 사용자 없이 유용한 정보를 제공 합니다.  
  
-   **항상** 를 일부 또는 사용자에 과부하가 하지 않는 앰비언트 표시기의 위치입니다. 앰비언트 표시기는 방식으로 위치를 지정할 수 없는 경우 다른 솔루션을 고려 합니다.  
  
-   **항상** 앰비언트 표시기를 관련 된 내용에 최대한 가깝게 배치 합니다.  
  
-   **항상** 사용할 수 있는 정보를 요약 하는 표시기를 만들려고 시도 합니다. 개수는 사용 가능한 데이터 항목 (예를 들어 "3" 대신 참조 하기만 하면 "참조")를 제공 하는 것이 좋습니다 또는 데이터를 요약 하려면 다른 방법으로 생각할 수 있습니다.  
  
    -   경우 여기서 표시기에 대 한 일 수 없으며 항상 계산을 표시, 즉시 고려 계산 되 고 값을 점진적 피드백을 제공 합니다. 예를 들어 변경 내용을 읽지 않은 메일 증가 수로 Windows Phone 전자 메일 라이브 타일을 새로 고칩니다 방식과 유사 하 게 사용 가능한 데이터에 대 한 업데이트를 반영 하는 애니메이션을 적용 하는 것이 좋습니다.  
  
-   **하지** 사용자 콘텐츠의 특정된 부분에서 합리적으로 수행할 수 있는 것 보다 더 많은 표시기를 추가 합니다. 앰비언트 표시기 사용자의 개입 없이 유용 합니다. 표시기는 오버플로 및 보기에 맞게 하는 다른 관리 제어를 필요로 할 경우의 외계를 손실 됩니다.  
  
#### <a name="gestures"></a>제스처  
 마스터 콘텐츠에 초점을 유지 하기 위해 사용자의 중요 한 측면은 오른쪽 제스처를 열고 추가 세부 정보 콘텐츠 해제를 지원 하 여 합니다.  
  
-   **항상** 사용자 추가 콘텐츠를 열려는 일부 명시적 제스처를 수행 해야 합니다. 일반적인 열려 제스처는 다음과 같습니다.  
  
    -   **: Hover** 도구 설명 또는 비 대화형 정보 콘텐츠  
  
    -   **명시적 명령을:** 인라인 프리젠터  
  
    -   **앰비언트 표시기를 두 번 클릭:** CodeLens 팝업 창  
  
-   **항상** Esc 키를 누를 때마다 세부 콘텐츠를 해제 합니다.  
  
-   **항상** 컨텍스트 개체에 UI 고려 합니다. 컨테이너 내에서 상호 작용을 허용 하는 콘텐츠 발표자 신중 하 게 하 고려 하는 사용자의 워크플로에 영향을 받을 가능성이 가리키기, 추가 정보를 표시 합니다.  
  
-   **하지** 가리키기 편집할 수 있는 것 같습니다. 또는 사용자 상호 작용을 초대 하는 콘텐츠를 표시 합니다. 이 동작은 도구 설명에 대 한 표준 동작은 커서가 있는 경우 더 이상 커서 생성 한 콘텐츠를 즉시 해제 하는 것 처럼 세부 콘텐츠 위로 커서를 이동 하 려 할 경우 사용자가 불편을 줄 수 없습니다.  
  
##  <a name="BKMK_SelectionModels"></a> 모델 선택  
  
### <a name="overview"></a>개요  
 선택 모델은 나타내고 사용자 인터페이스 내에서 필요한 하나 이상의 개체에 대 한 작업을 확인 하는 데 사용 하는 메커니즘입니다. 이 항목에서는 Visual Studio 문서 편집기 내에서 선택 상호 작용 패턴: 텍스트 편집기, 디자인 화면 및 모델링 합니다.  
  
 사용자가 어떤 작업 중인, Visual Studio로 나타낼 수 있어야 합니다. 및 Visual Studio로 응답 해야 예측 가능한 방식으로 피드백 작동에 대 한 사용자에 게 합니다. 차이점 또는 사용자와 사용자 인터페이스 간의 드려 되지 수 있는 작업을 검색 하는 사용자 발생할 수 있습니다 의도 하지 않은 결과입니다. 종종 없거나 항목 또는 변경 된 사용자에 게 표시 될 때까지 오류 인지 되 합니다. 따라서 선택 모델은 사용자 인터페이스 디자인의 가장 중요 한 부분 중 하나입니다. Visual Studio에서 선택 모델이 Windows와 일치 하지만에 약간의 변형입니다.  
  
 Windows 에서처럼 Visual Studio에서 선택 모델 상호 작용 발생 하는 컨텍스트에 따라 달라 집니다. 선택 항목은 네 가지 유형의 개체에서 발생할 수 있습니다.  
  
-   텍스트  
  
-   그래픽 개체  
  
-   목록 및 트리  
  
-   표  
  
 이러한 개체에는 세 가지 유형의 선택  
  
-   연속  
  
-   서로 분리 된  
  
-   Region  
  
#### <a name="scope"></a>범위  
 선택 항목의 가장 중요 한 구성 (활성화) 어느 창에 작업할 수 있고 포커스가 있는 (선택) 사용자가 알고 있는지 확인 하는 합니다. Visual Studio가 windows에서 창 관리 기능을 확장 하는데 정품 인증 체계는 동일: 창에 포커스 창와 상호 작용 합니다. Visual Studio에는 정품 인증에 대 한 두 개의 표시기: 문서 창 및 도구 창에 대 한 합니다.  
  
 문서 창에 대 한 활성 창이 앞에 오는 배경 색을 변경 하는 문서 창의 탭으로 표시 됩니다.

 도구 창에 대 한 활성 창 제목 표시줄 영역 도구 창의 색이 변경으로 표시 됩니다.  
  
 ![Visual Studio에서 활성 도구 창 선택](../../extensibility/ux-guidelines/media/0713-02_activetoolwindow.png "0713 02_ActiveToolWindow")

 **노드의 latent 선택 영역을 보여 주는 비활성 도구 창**  
  
 창이 옵션이 활성화 되 면 집중적이이 섹션의 지침에 설명 된 선택 모델에 따라 표시 됩니다.  
  
#### <a name="context"></a>컨텍스트  
 Visual Studio는 사용자가 작업 하는 위치를 추적 하는 컨텍스트의 강력한 개념을 유지 하도록 설계 되었습니다. 도구 또는 문서 창 인지 창이 하나만 활성화 되어. 그러나 최상위 문서 창 latent 선택 항상 유지합니다. 도구 창에 포커스가 있을 수 있고 수, 마지막 활성 이었던 문서 창의 선택 항목을 비활성 상태로에 표시 합니다. 이 문서는 편집을 반환 하 고 도구 창과 문서 창 사이의 원활 하 게 이동할 수 있도록 Visual Studio의 상태를 유지에 자습서에서 사용자의 컨텍스트를 유지 하도록 수행 됩니다.  
  
### <a name="text-selection"></a>텍스트 선택  
 에 설명 된 모양 및 동일한 텍스트 선택 모델을 사용 하는 기본 제공 텍스트 편집기와 같은 텍스트 엄격 하 게 되는 visual Studio 편집기는 [마우스 및 포인터](https://msdn.microsoft.com/en-us/library/dn742466.aspx) 에 Windows 사용자 환경 상호 작용 지침의 페이지 MSDN 합니다. 텍스트 편집기에서 입력된 포커스가 삽입 지점 세로 막대로 표시 됩니다. 에서는 삽입 지점이 굵은 하 고 그 뒤에 표시 무엇이 든의 역으로 색이 지정 된 단일 픽셀입니다. 으로 설정 하는 속도 따라 깜박이 **커서 깜박임 속도** 에서 설정는 **속도** 탭은 **키보드** 제어판 애플릿에서 합니다.  
  
#### <a name="contiguous-and-disjoint-selection"></a>연속 및 비연속 선택  
 텍스트 편집기에서 선택은 연속만 제공 합니다. 비연속 텍스트 선택 항목 허용 되지 않습니다 되지만 그래픽 개체 편집기에서 해결 해야 합니다. 사용자의 마우스 포인터가 텍스트 영역 위에, 커서가 i 형으로 바뀝니다. 한 번의 클릭 클릭 위치에 텍스트 편집기에서 다음 삽입 지점을 넣습니다. 마우스 단추를 누른 상태로 강조 표시 선택 영역이 시작과 마우스 단추에서 손을 떼기 끝을 강조 표시 선택 영역이 있습니다.  
  
#### <a name="region-selection-box-selection"></a>지역 선택 (상자 선택)  
 Visual Studio 텍스트 편집기에서 지역 선택 항목을 지원 하 고이 상자 선택 이라고 합니다. 상자 선택 일반 텍스트 스트림에 따르지 않는 텍스트 영역을 선택할 수 있습니다. 표준 텍스트 선택와 마찬가지로 선택이 연속적 이어야 합니다. Alt 키를 누른 채 마우스를 사용 하 여 끌거나 상자 선택 시작 됩니다. Alt 키 및 Shift 키를 누른 채 화살표 키를 사용 하 여 선택 영역을 나타내기 위해 여 상자 선택을 초기화할 수도 있습니다. 상자 선택 일반 선택 강조 표시를 사용 하 고 삽입 지점 커서 선택 영역 끝 깜박입니다.이 보여 줍니다.  
  
 ![국가별 &#40;상자&#41; Visual Studio에서 선택](../../extensibility/ux-guidelines/media/0713-04_boxselection.png "0713 04_BoxSelection")  
  
 **Visual Studio에서 지역 (상자) 선택**  
  
#### <a name="text-selection-appearance"></a>텍스트 선택 항목 모양  
 편집기에서 활성 및 비활성 선택에 사용 되는 색을 사용자 지정할 수 있습니다. 편집기의 시각적 모양을 사용자 지정 사용자로 이동 하 **도구 > 옵션**, 한 다음 **환경 > 글꼴 및 색 > 텍스트 편집기**합니다.  
  
### <a name="graphical-selection"></a>그래픽 선택  
  
#### <a name="interaction"></a>상호 작용  
 그래픽 개체 선택 복잡 하 고 다양 한 요인에 따라 달라 집니다.  
  
-   **편집기의 기본 선택 모델입니다.** 그래픽 개체를 포함 하는 편집기는 텍스트 또는 표를 편집 하려면 데도 사용할 수 있습니다. 예를 들어 편집기에서는 Visual Studio XAML 디자이너와 같은 그래픽 개체의 배치 하는 텍스트 기반 편집기 수도 있습니다. 여러 개체 유형을 지 원하는 사용자에서 다른 유형의 개체가 구성 하는 그룹을 선택 하는 방법에 발생할 수 있습니다.  
  
-   **기본 및 보조 선택 상태를 지원 합니다.** 편집기 상태 함께, 개체를 편집할 수 있도록 서로 맞춰져, 함께, 크기를 조정 하는 기본 및 보조 선택 영역을 제공할 수 있습니다.  
  
-   **내부 편집 지원 합니다.** 편집기에 그 그래픽 개체를 편집할 수의 콘텐츠를 사용할 수 있습니다. 예를 들어 사각형은 사용자가 변경할 수 있는 내부에 텍스트를 포함할 수도 있습니다. 또한 해당 텍스트 가운데 또는 양쪽 맞춤 수 없습니다. 내부 편집 보다 세부적인된 수준의 사용자 상호 작용을 포함를 하므로 적절 한 시각 신호 집합이 사용자에 게 상태 정보를 제공 합니다.

|선택 핸들을 가리키도록|포인터 크기 조정 커서를 변경 합니다. 를 회전을 지원 하는 개체에 대 한 일부 선택 핸들 변경 될 수 있습니다 포인터 회전 커서를 포인터 선택 핸들에 대해 다른 방식 (예: 멀리 이동) 배치 됩니다.|

|개체 선택|테두리, 점선 또는 개체의 경계를 강조 표시 하 게 처리 다른 시각적 구분으로 표시 됩니다.|

#### <a name="keyboard-interaction"></a>키보드 상호 작용  
  
|입력|결과|  
|-----------|------------|  
|탭|편집기에서 개체의 논리적 순서 간의 포커스 표시기를 이동합니다. 왼쪽에서 오른쪽으로 또는 위쪽에서 아래쪽에 따라 수도 **TabIndex** 하거나 해당 속성 값, 개체 만들기 순서 및 편집기의 전반적인 용도입니다. Shift + Tab 포커스 표시기의 방향을 반대로 바꿉니다.|  
|스페이스바|키 입력은 유지 관리 하는 동안 패닝 모드를 활성화 합니다. 추가 마우스 입력은 뷰포트 위치를 이동 해야 합니다.|  
|Ctrl+스페이스바|키 입력은 유지 관리 하는 동안에 확대/축소 모드를 활성화 합니다. 추가 마우스 입력을 확대/축소 비율을 늘리거나 줄이는 필요 합니다.|  
|Ctrl + Alt + 빼기 기호|한 수준으로 확대/축소 비율을 줄입니다.|  
|Ctrl + Alt + 더하기 기호|한 수준으로 확대/축소 비율을 늘립니다.|  
|Shift 또는 Ctrl|선택 영역 그룹에 개체를 추가합니다. 또한 Ctrl을 사용 하면 선택 그룹에서 개체를 개별적으로 제거 수 있습니다.|  
|Enter 키|개체에 대 한 기본 명령을 수행 (일반적으로 열어 또는 편집).|  
|F2|개체에 대 한 내부 편집을 활성화 합니다.|  
|화살표 키|화살표 키를 누르면, 조금씩 (예를 들어, 한 번에 1 픽셀)의 방향으로 선택된 된 개체를 이동|  
|Ctrl + 화살표 키|화살표 키를 누르면, 단위 이동 (예를 들어, 한 번에 10 픽셀)의 방향으로 선택된 된 개체를 이동|  
|Shift + 화살표 키|해당 방향 (예를 들어, 한 번에 1 픽셀) 작은 단위로 선택된 된 개체 크기를 조정합니다|  
|Ctrl + Shift + 화살표 키|단위 이동 (예를 들어, 한 번에 10 픽셀) 해당 방향으로 선택된 된 개체 크기를 조정합니다|  
  
 사용자가 현재 위치에서에서 컨트롤을 편집 하는 경우 좋을 수 있습니다를 자동으로 조정 사용자 입력을 사용 하는 개체에 대 한 합니다. 예를 들어 사용자가 레이블 컨트롤을 편집, 레이블을 사용자가 방금 입력 한 텍스트를 표시 하 증가 하도록 합니다. 이렇게 하지 않으면 사용자를 조정 해야 컨트롤 수동으로 텍스트를 편집한 후입니다. 사용자가 컨트롤의 많은 경우이 기계적 및 작업 생산성이 저하 됩니다.  
  
#### <a name="graphical-containers"></a>그래픽 컨테이너  
 경우에 따라 그래픽 편집기는 Windows Forms Panel 컨트롤 또는 HTML 디자이너에서 모눈 레이아웃 컨트롤과 같은 다른 그래픽 개체에 대 한 컨테이너를 제공합니다. 편집기에서는 그래픽 다른 개체에 대 한 컨테이너를 제공 하는 경우에 다음과 같은 선택 모델의 컨테이너 (표준 모델을 위에서 설명한 컨테이너 팔 로우 내의 개체)에 대 한 사용 해야 합니다.  
  
|입력|결과|  
|-----------|------------|  
|컨테이너에 대 한 클릭|직접 포함 된 개체 중 하나를 선택 하지 않고 컨테이너 개체를 선택 합니다. 컨테이너 이동 하거나 표준 마우스 및 키보드 입력 (위에서 설명한)와 함께 크기 조정 수 있습니다. 컨테이너에 관계에 포함 된 개체를 이동 하지만 포함 된 개체도 직접 선택 하지 않으면 크기가 조정 되지 않습니다.|  
|컨테이너의 경계 영역 위로 마우스를 가져가고합니다|컨테이너를 이동할 수 있는지를 나타내는 이동 커서로 마우스를 설정 합니다.|  
|컨테이너의 경계 영역으로 끌어|마우스 이동 커서로 변경 하 고 컨테이너 (및 내에서 포함 된 개체)를 이동 합니다. 첫 번째 없이 컨테이너를 이동할 수 없습니다 한 번의 클릭으로 선택 합니다.|  
|컨테이너 내 개체에 대 한 클릭|(선택) 경우 컨테이너를 선택 취소 하 고 클릭 한 개체를 선택 합니다.|  
|Shift + 클릭 OR Ctrl + 포함 된 개체 및/또는 컨테이너에서를 클릭합니다|기존 선택 또는 선택 영역 그룹을 클릭 한 개체를 추가합니다. 클릭 한 개체가 선택 영역 그룹의 멤버인 이미 있으면 선택 그룹에서 제거 됩니다.|  
  
 포함 된 개체는 이전 섹션에 설명 된 대로 기본 선택 모델 준수 해야 합니다. Windows Forms 디자이너의 유용성 테스트를 통해 사용자는 포함 된 개체에 대 한 원활한 액세스 (containment 개체에 의해 적용 됨) 하는 중간 단계 없이 필요 합니다.

#### <a name="tree-view"></a>트리 뷰  
 트리 보기의 선택 항목으로 간단한 강조 표시 됩니다. 노드 이름 또는 노드 아이콘을 클릭 하는 경우 노드가 선택 됩니다. 삼각가 중 노드의 왼쪽 문자 모양이 확장 또는 축소 트리 컨트롤 있지만 예외적으로 사용자의 선택에 영향을 주지 않습니다: 부모에 하나씩 선택 시 해당 노드의 자식 항목을 선택 하면 부모 노드를 축소 합니다.  
  
 ![Visual Studio에서의 일반적인 트리 뷰](../../extensibility/ux-guidelines/media/0713-25_treeview.png "0713 25_TreeView")  
  
 **Visual Studio에서의 일반적인 트리 뷰**  
  
 트리 뷰는 트리의 여러 수준 간에 연속 및 비연속 선택을 지원할 수 있습니다. 인접 하거나 서로 분리 된 여러 선택 항목 표시 되는 트리 노드에서 수행 해야 합니다. 노드를 축소 하는 경우에 비연속 선택 손실 되 고 축소 된 노드 선택 영역을 가져옵니다. 이러한 방식으로 영향을 받을 작업에서 노드를 볼 수 있습니다. 노드가 축소 되 됩니다 명확 하지 않은 노드를 영향을 줄 수 있습니다.  
  
 부모 노드가 선택 되 면 부모 및 모든 자식에 적용 하는 작업에 대 한 적절 한 사례 수 있지만 부모에 게 작업을 적용 해야 합니다. 이 경우 확인란 또는 사용자에 게 "모든 자식 항목에 적용" 옵션을 명시적 하 확인 대화 상자와 같은 작업을 하는 동안 추가 UI를 제공 합니다.  
  
##### <a name="renaming"></a>이름 바꾸기  
 트리 노드 이름 바꾸기를 지원, 이름 바꾸기 위치에서 수행 해야 합니다. 내부 작업에는 Visual Studio의 모든 트리 컨트롤에서 표준 이어야 합니다. 사용자 입력을 허용할 준비가 노드를의 전체 이름을 포함 텍스트 선택 항목으로 즉시 내부 편집 모드를 활성화 하는 이름 바꾸기 명령을 제공 합니다. 노드는 파일을 나타내는 파일 이름 확장명을 포함 해야 합니다. 강조 표시 선택 영역이 파일 이름 및 확장명이 아닌의 본문만을 포함 되어야 합니다.  
  
|입력|결과|  
|-----------|------------|  
|Enter 키|이름 바꾸기 작업을 커밋합니다.|  
|Esc 키|이름 바꾸기 작업을 취소합니다.|  
|내부 편집 영역 바깥쪽을 클릭 하면|이름 바꾸기 작업을 커밋합니다.|  
|실행 취소|이름 바꾸기 작업을 취소 하는 쉬운 실행 취소를 제공 합니다.|  
  
#### <a name="selection-within-lists-and-grid-controls"></a>목록 및 그리드 컨트롤 내에서 선택  
 목록 선택의 주요 개념 행 기반으로 하나의 단위로 선택 전체 행 하도록 선택할 때 사용자에 게 즉 됩니다. 반면, 표는 특정 셀의 행의 다른 측면에 영향을 주지 않고 선택 허용할 수 있습니다. 표는 중첩 된 행 (예:는 트리 모눈) 계층 구조에 선택 된 부모 행 상호 작용 하 여 선택이 취소의 전체 분기를 허용 하는 계층 구조를 포함할 수도 있습니다. 선택 목록에는 데이터의 행 전체에서 단순 강조 색으로 표시 됩니다. 포커스를 현재 편집 가능한 행 또는 셀 (모든 셀은 읽기 전용으로 설정 하는 경우에 행) 주위에 단일 픽셀 점선된 테두리를 표시 합니다.  
  
> [!NOTE]
>  **포커스** 및 **선택** 은 서로 다른 개념입니다. *포커스* 요소는 UI의 대상 표시 하지만 입력을 받는 다른 개체에서 명시적으로 지시를 *선택* 상태가 된 이후에 개체의 집합 개체의 포함 여부의 상태를 나타냅니다 작업을 수행할 수 있습니다.  
  
 인접 한,이 목록에서 선택 될 수 있습니다 또는 지역입니다. 경우 여러 선택 항목은 허용, 연속을 비연속 선택 해야 항상 지원 될 지역 (상자) 선택 항목에 대 한 지원 하는 동안에 선택 사항입니다. 지역 선택 항목의 목록 본문의 빈 공간으로 끌어 열의 시작 됩니다.  
  
|Object|선택|  
|------------|---------------|  
|목록|연속|(다중 선택을 사용할 수 있음) 하는 경우 항상 지원 됩니다.|  
|목록|서로 분리 된|(다중 선택을 사용할 수 있음) 하는 경우 항상 지원 됩니다.|  
|목록|Region|선택 사항 지원 합니다. 목록 본문의 공백에 마우스를 끌어 시작.|  
  
 목록에 두 번 클릭 하면 클릭이 발생 하는 행을 선택 합니다. 사용자 목록 셀 내부 편집을 지 원하는 경우 셀 내부 편집에 대 한 즉시도 활성화 됩니다. 그렇지 않으면 전체 행 선택 즉시 되 고 강조 표시를 보여 줍니다.  
  
 목록 본문에서 다음 세 가지 중 하나를 수행 합니다.  
  
-   목록에서 지 원하는 마우스 아래쪽에 있는 경우 공백 지역 선택 시작  
  
-   목록 셀 또는 행 끌기 소스 되 고 지원 되 면 끌기/놓기 작업을 시작 합니다.  
  
-   현재 행을 선택  
  
##### <a name="in-place-editing"></a>내부 편집  
 내부 편집이 허용 되 면 두 가지 기본 모델: 약간만 편집 컨트롤 및 속성 선택 합니다. 간단한 편집 컨트롤과 콘텐츠 강조 표시 되 면 사용자가 활성화 내부 편집 되는 즉시 입력에 대 한 합니다. 속성 선택 구현 되는 경우 여기서 내부 편집 모드를 활성화 하 고 현재 선택 영역을 강조 표시 되지 않은 속성 선택을 호출 하는 단추가 표시 됩니다. 선택기 단추 셀에 오른쪽 맞춤 여야 합니다. 내부 편집 예 참조는 **속성 창** 및 **작업 목록** Visual Studio에서.  
  
##### <a name="keyboard-support"></a>키보드 지원  
 키보드 지원 목록과 격자의 선택에 대 한 표준 Windows 규칙을 따릅니다.

-   Shift + 화살표 방향 화살표 키도 연속적인 선택을 수행 합니다.  
  
-   Ctrl + 화살표 사이에서 추가 및 제거 목록 항목 선택 영역을 만드는 비연속 선택 스페이스바 토글 옵니다.  
  
-   중첩 된 계층 구조를 포함 하는 표에 대 한 오른쪽 화살표 키 확장 부모 행와 왼쪽 화살표 키를 하나 축소 됩니다.  
  
-   Tab 키 셀을 편집할 수 있는 경우 현재 행의 셀 간에 포커스를 이동 합니다.  
  
-   Enter 키 목록에서 항목에 기본 명령을 수행 (종종 **열려**).  
  
-   F2 키 위치에 현재 선택 된 셀에 대 한 편집을 활성화 합니다.

##  <a name="BKMK_TouchInput"></a> 터치식 입력

### <a name="levels-of-experience"></a>환경의 수준  
 환경에서는 다음과 같은 수준의 터치에 투자 관심 원하는 수준에 따라 터치 기능을 제공 하는 의사 결정 지침으로 사용 하는 데 사용 됩니다.  
  
-   **기본 환경을** 를 제공할 팀 터치 기능 작업 전체에서 배달 못 한 ends가 없기 때문입니다.  
  
-   **환경에 최적화 된** 가장 (예를 들어 일반적으로 사용할 수 있는 인터넷 브라우저 응용 프로그램에서) 일반적인 터치 기능을 제공 하는 팀입니다.  
  
-   **경험 상승** 제스처 마찬가지로 이러한 기능을 추가 하는 팀의 되었거나 응용 프로그램을 만들 수 있는 기타 선택적 기능 터치 우선 친숙 한 이름입니다.

#### <a name="gestures"></a>제스처  
 제스처 더 복잡 한 상호 작용을 필요할 수 있는 명령에 바로 가기를 사용자를 제공 합니다. Windows 지침에서 참조 [데스크톱 응용 프로그램에 대 한 일반적인 터치 제스처](http://msdn.microsoft.com/en-us/library/windows/desktop/dd940543\(v=vs.85\).aspx), 이동 및 확대/축소 등의 간단한 제스처를 비롯 한 대부분 제스처에 대 한이 지침을 따릅니다.
