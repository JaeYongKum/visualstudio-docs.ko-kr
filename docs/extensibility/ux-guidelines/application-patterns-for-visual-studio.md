# Visual Studio에 대 한 응용 프로그램 패턴
[!INCLUDE[vs2017banner](../../code-quality/includes/vs2017banner.md)]

##  <a name="a-namebkmkwindowinteractionsa-window-interactions"></a><a name="BKMK_WindowInteractions"></a> 창 상호 작용  
  
### <a name="overview"></a>개요  
 두 개의 주 창 Visual Studio에서 사용 되는 유형은 문서 편집기와 도구 창입니다. 드물지만 가능한 큰 모덜리스 대화 상자는. 이러한 옵션은 모든 셸에서 모덜리스, 해당 패턴은 근본적으로 다릅니다. 이 항목에서는 문서 창, 도구 창 및 모덜리스 대화 상자 간의 차이점을 설명 합니다. 모달 대화 패턴에 설명 합니다 [대화 상자의](../../extensibility/ux-guidelines/application-patterns-for-visual-studio.md#BKMK_Dialogs)합니다.  
  
### <a name="comparing-window-usage-patterns"></a>창 사용 패턴을 비교합니다.  
 **문서 창** 거의 항상 내에 표시 되는 문서도 합니다. "Center 단계" 문서 편집기 이렇게 하면 추가 도구 창 주위 정렬 합니다.  
  
 A **도구 창** 으로 IDE의 가장자리에 대해 별도, 더 작은 창 표시, 숨김, 또는 자동 숨김 수 –를 축소 되어 가장 자주 표시 됩니다. 그러나 때로는 제공 하는 물론 문서 내에서 선택을 취소 하 여는 **창/도킹** 창에는 속성입니다. 이 인해 더 하지만 또한 일반적인 설계 결정 사항: Visual Studio에 통합 하는 동안, 기능 도구 창과 문서 창에 표시 되는지 여부를 결정 해야 합니다.  
  
 **모덜리스 대화 상자** 없는 Visual Studio에서 것이 좋습니다. 많은 부분 부동 정의 –는-도구 창을 하며 이와 같이 구현 합니다. 모덜리스 대화 상자는 셸의 쪽에 도킹 된 일반 도구 창의 크기는 너무 제한 하는 경우에 사용할 수 있습니다. 사용자 대화 상자를 보조 모니터로 이동 될 수 없는 경우에 사용할 수 있습니다.  
  
 필요 하다 고 생각 신중 하 게 컨테이너 유형에 대 한 합니다. UI 디자인에 대 한 일반적인 사용 패턴 고려 사항에서 다음 테이블에 있습니다.

##  <a name="a-namebkmktoolwindowsa-tool-windows"></a><a name="BKMK_ToolWindows"></a> 도구 창  
  
### <a name="overview"></a>개요  
 도구 창은 문서 창에서 발생 하는 사용자의 작업을 지원 합니다. 이러한 Visual Studio를 제공 하 고 조작할 수 있는 기본 루트 개체를 나타내는 계층 구조를 표시 하는 데 사용 될 수 있습니다.  
  
 IDE의 새로운 도구 창을 고려할 때 작성자 수행 해야 합니다.  
  
-   작업에 적합 한 기존 도구 창을 사용 하 여 유사한 기능을 가진 새로운 레코드를 만들지 않으며 새 도구 창은 비슷한 "도구" 또는 유사한 창으로 또는 기존 창을 피벗 허브로 설정 하 여 통합 될 수 없는 기능을 제공 하는 경우에 생성 해야 합니다.  
  
-   도구 창의 위쪽에 필요한 경우에 표준 명령 모음을 사용 합니다.  
  
-   컨트롤 프레젠테이션 및 키보드 탐색을 위한 다른 도구 창에 이미 있는 패턴과 일치 수 있습니다.  
  
-   다른 도구 창에서 컨트롤 표시와 일치 됩니다.  
  
-   문서 관련 도구 창을 표시할지 자동-가능한 경우 부모 문서를 활성화 하는 경우에 표시 되도록 합니다.  
  
-   키보드 (지원 화살표 키)를 사용 하는 탐색 가능한 창 콘텐츠를 확인 합니다.  
  
#### <a name="tool-window-states"></a>도구 창 상태  
 Visual Studio 도구 창 있는 일부 (예: 자동 숨기기 기능) 사용자 활성화는 다양 한 상태입니다. 와 같은 기타 상태 자동 표시 도구 창을 올바른 컨텍스트에서 표시 및 숨기기 필요 하지 않은 경우 하도록 허용 합니다. 총에서 5 개의 도구 창 상태 있습니다.  
  
-   **고정/고정** 도구 창과 문서 영역의 네 면 중 하나에 연결할 수 있습니다. 도구 창의 제목 표시줄에 압정 아이콘이 나타납니다. 도구 창을 도킹할 수 있는 가로 또는 세로로 셸 및 기타 도구 창을의 가장자리를 따라 및 탭 연결 될 수도 있습니다.  
  
-   **자동 숨김** 도구 창을 고정이 해제 됩니다. 창을 보이지 문서 영역의 가장자리에 (사용 하 여 탭의 도구 창 이름과 해당 아이콘)을 그대로 두고 슬라이드 할 수 있습니다. 사용자의 탭 위로 가져갈 때 도구 창을 이동 하면서 합니다.  
  
-   **자동 표시** 는 편집기와 같은 UI의 다른 부분 시작 되거나 포커스를 획득 하는 경우에 자동으로 도구 창을 표시 합니다.  
  
-   **부동** 도구 창을 IDE 외부에서 마우스를 이동 합니다. 이것은 다중 모니터 구성이 적합 합니다.  
  
-   **탭된 문서** 도구 창과 문서 내에서 잘 도킹 될 수 있습니다. 허용 되는 프레임의 가장자리에 도킹 보다 더 많은 부동산을 필요로 하는 개체 브라우저와 같은 큰 도구 창에 유용 합니다.

#### <a name="single-instance-and-multi-instance"></a>단일 인스턴스 및 다중 인스턴스  
 도구 창에는 단일 인스턴스 또는 다중 인스턴스입니다. 다중 인스턴스 도구 창을 수도 있지만 일부 단일 인스턴스 도구 창을 현재 문서 창와 관련 될 수 있습니다. 다중 인스턴스 도구 창을 창의 새 인스턴스를 만들어 창/새 창 명령에 응답 합니다. 다음 그림에서는 창의 인스턴스 활성화 되어 있을 때 설정 하는 새 창 명령 도구 창을 보여 줍니다.  
  
 ![Visual Studio의 명령을 사용하는 도구 창](../../extensibility/ux-guidelines/media/0702-02_toolwindowenablingcommand.png "0702-02_ToolWindowEnablingCommand")  
  
 **창의 인스턴스 활성화 되어 있을 때 ' 새 창 ' 명령을 사용 하는 도구 창**  
  
 단일 인스턴스 도구 창은 숨기 거 나 다중 인스턴스 도구 창을 수 수 닫을 뿐만 아니라 숨겨진 동안 표시 되 될 수 있습니다. 탭 연결, 부동 또는 다중 문서 MDI (인터페이스) 자식 창 (문서 창으로와 유사)으로 설정 모든 도구 창은 도킹 될 수 있습니다. 모든 도구 창을 창 메뉴에서 적절 한 창 관리 명령에 응답 해야 합니다.

#### <a name="document-specific-tool-windows"></a>문서 관련 도구 창  
 일부 도구 창의 지정 된 형식의 문서를 기준으로 변경 하도록 설계 되었습니다. 이러한 창을 IDE에서 현재 문서 창에 적용할 수 있는 기능을 반영 하기 위해 지속적으로 업데이트 합니다.  
  
 도구 창 내용이 선택한 편집기를 반영 하도록 변경의 예로 도구 상자 및 문서 개요. 편집기 창에는 컨텍스트를 제공 하지 않는 포커스가 있을 때 이러한 창의 워터 마크를 보여 줍니다.  
  
#### <a name="navigable-list-tool-windows"></a>탐색 가능한 목록 도구 창  
 일부 도구 창에는 사용자와 상호 작용할 수 탐색 가능 항목의 목록을 표시 합니다. 이 유형의 창에 항상 있어야 목록에서 현재 항목에 대 한 피드백 경우 창이 활성화 되는 경우에 있습니다. 목록에 응답 하는 **GoToNextLocation** 및 **GoToPrevLocation** 도 창에서 현재 선택 된 항목을 변경 하 여 명령  
  
 탐색 가능한 목록 도구 창의 예로 솔루션 탐색기 및 찾기 결과 창입니다.

##  <a name="a-namebkmkdocumenteditorconventionsa-document-editor-conventions"></a><a name="BKMK_DocumentEditorConventions"></a> 편집기 도움말 표기법  
  
### <a name="document-interactions"></a>문서 상호 작용  
 "문서 도" IDE 내에서 가장 큰 공간 이며 여기서 사용자 일반적으로 다루었습니다 알리고 추가 도구 창에서 지 원하는 해당 작업을 완료 하기 위해 합니다. 문서 편집기에는 사용자가 열리고 Visual Studio 내에서 저장 하는 작업의 기본 단위를 나타냅니다. 솔루션 탐색기 또는 다른 활성 계층 구조 창에 연결 된 선택 항목에 대 한 강력한 개념이 유지 합니다. 사용자에 이러한 계층 구조 창 중 하나를 가리키고 솔루션, 프로젝트 또는 Visual Studio 패키지에서 제공 하는 다른 루트 개체에는 문서가 포함 된 위치와 관계를 알고 있어야 합니다.  
  
 문서 편집 일관적인 사용자 경험이 필요합니다. 사용자는 작업에 대신 찾기 명령 및 창 관리에 집중할 수 있도록 해당 문서 유형을 편집 하기 위한 사용자 작업에 가장 잘 맞는 문서 보기 전략을 선택 합니다.  
  
#### <a name="common-interactions-for-the-document-well"></a>잘 문서에 대 한 일반적인 상호 작용  
  
-   공통에서 일관 된 상호 작용 모델을 유지 관리 **새 파일** 및 **파일 열기** 발생 합니다.  
  
-   문서 창이 열리면 관련된 창 및 메뉴의 관련된 기능을 업데이트 합니다.  
  
-   메뉴 명령에 공통적인 메뉴와 같은 통합 적절 하 게 **편집**, **형식**, 및 **보기** 메뉴. 상당한 양의 특수 한 명령 사용할 수 있는 새 메뉴 표시 되는 문서에 포커스가 경우에 만들 수 있습니다.  
  
-   포함된 된 도구 모음 편집기의 위쪽에 배치 될 수 있습니다. 이 편집기 외부에서 표시 되는 별도 도구 모음을 지정 하는 것이 좋습니다.  
  
-   솔루션 탐색기 또는 유사한 액티브에서 선택한 항목을 항상 유지 계층 구조 창입니다.  
  
-   솔루션 탐색기에서 문서를 두 번 클릭 하면 동일한 작업을 수행 해야 **열려**합니다.  
  
-   재정의 하거나 기본 작업에 사용 하 여 특정된 문서 형식을 다시 설정 하려면 사용자 있어야 문서 형식에 둘 이상의 편집기를 사용할 수 있는 경우는 **연결 프로그램** 파일을 마우스 오른쪽 단추로 클릭 하 고 선택 하 여 대화 상자 **Open With** 바로 가기 메뉴에서입니다.  
  
-   문서에는 마법사를 제대로 작성 하지 마세요.  
  
### <a name="user-expectations-for-specific-document-types"></a>특정 문서 유형에 대 한 사용자의 기대  
 여러 가지 기본 유형의 문서 편집기 및 각 동일한 형식의 다른 항목과 일치 하는 상호 작용 집합이 있습니다.  
  
-   **텍스트 기반 편집기:** 코드 편집기, 로그 파일  
  
-   **디자인 화면:** WPF 폼 디자이너, Windows forms  
  
-   **대화 상자 스타일 편집기:** 매니페스트 디자이너, 프로젝트 속성  
  
-   **모델 디자이너:** 워크플로 디자이너, codemap, 아키텍처 다이어그램, 진행  
  
 잘 문서를 사용 하 여 비 편집기 형식이 몇 가지 있습니다. 문서 자체를 편집 하지은, 동안 문서 창에 대 한 표준 상호 작용에 따라 필요가 있습니다.  
  
-   **보고서:** IntelliTrace 보고서, Hyper-v 보고서 프로파일러 보고서  
  
-   **대시보드:** 진단 허브  
  
#### <a name="text-based-editors"></a>텍스트 기반 편집기  
  
-   문서를 열지 않고도 문서를 미리 볼 수 있도록 하는 미리 보기 탭 모델에 참여 합니다.  
  
-   문서 개요와 같은 도우미 도구 창 내에서 문서 구조를 표시할 수 있습니다.  
  
-   IntelliSense (필요한 경우)는 다른 코드 편집기로 일관 되 게 작동 합니다.  
  
-   팝업 창이 나 보조 UI CodeLens와 같은 기존 유사한 UI에 대 한 유사한 스타일 및 패턴을 따릅니다.  
  
-   상태 표시줄에 또는 문서 맨 위에 있는 정보 표시줄 컨트롤에서 문서 상태에 대 한 메시지가 표시 됩니다.  
  
-   사용자가 글꼴 및 색을 사용 하 여 모양을 사용자 지정할 수 있어야는 **도구 > 옵션** 공유 글꼴 및 색 페이지 또는 하나의 특정 편집기 페이지입니다.  
  
#### <a name="design-surfaces"></a>디자인 화면  
  
-   빈 디자이너는 시작 하는 방법을 나타내는 화면에 워터 마크를 사용 해야 합니다.  
  
-   보기 전환 메커니즘을 코드 편집기 또는 양쪽 창와의 상호 작용 허용 문서 창 내에서 탭을 열려면 두 번 클릭과 같은 기존 패턴을 따릅니다.  
  
-   디자인 화면에 요소를 추가 해야 도구 상자를 통해 매우 특정 도구 창을 필요한 경우를 제외 합니다.  
  
-   화면에서 항목 선택 일관 된 모델을 따릅니다.  
  
-   포함 된 도구 모음 같은 문서 관련 명령, 일반적인 명령이 포함 되어 **저장**합니다.  
  
#### <a name="dialog-style-editors"></a>대화 상자 스타일 편집기

-   편집기 내에서 탭 문서 탭의 모양을 일치 해서는 안, 허용 된 내부 탭 두 스타일 중 하나를 일치 하는지 확인 합니다.

-   디자인 화면에 요소를 추가 하는 도구 상자를 통해 수행 되어야 합니다.  
  
-   화면에서 항목 선택 일관 된 모델을 따릅니다.  
  
-   포함 된 도구 모음 같은 문서 관련 명령, 일반적인 명령이 포함 되어 **저장**합니다.

##  <a name="a-namebkmkdialogsa-dialogs"></a><a name="BKMK_Dialogs"></a> 대화 상자  
  
### <a name="introduction"></a>소개  
 일반적으로 하나의 개별 사용자의 작업 단위를 지원 해야 및 해제 한 다음 Visual Studio에서.  
  
 대화가 필요를 확인 한 확률이 높은 순서 대로 세 가지 선택 사용할 수 있습니다.  
  
1.  Visual Studio의 공유 대화 상자 중 하나에 기능을 통합 합니다.  
  
2.  기존 비슷한 대화에서 발견 한 패턴을 사용 하 여 고유한 대화 상자를 만듭니다.  
  
3.  새 대화 상자, 다음과 같은 상호 작용 및 레이아웃 지침을 만듭니다.  
  
 이 항목에서는 Visual Studio는 워크플로 내에서 해당 대화 패턴 및 대화 상자 디자인에 대 한 일반적인 규칙을 선택 하는 방법에 설명 합니다.  
  
### <a name="themes"></a>테마  
 Visual Studio에서 대화 상자는 두 가지 기본 스타일 중 하나를 따르십시오.  
  
#### <a name="standard-unthemed"></a>표준 (unthemed)  
 대부분의 대화 상자는 표준 유틸리티 대화 상자 이며 unthemed 이어야 합니다. 스타일이 적용 된 "최신" 단추나 컨트롤을 작성 하려고 하거나 re 템플릿 공용 컨트롤 하지 마십시오. 컨트롤 및 크롬 모양에 따라 [대화 상자에 대 한 표준 Windows 데스크톱 상호 작용 지침](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742499\(v=vs.85\).aspx)합니다.  
  
#### <a name="themed"></a>테마가 지정 된  
 전문 "서명" 대화 상자 테마를 적용할 수 있습니다. 테마가 지정 된 대화 상자는 스타일과 관련 된 몇 가지 특수 한 상호 작용 패턴에도 있는 고유 모양에 합니다. 테마 이러한 요구 사항을 충족 하는 경우에 대화 상자:  
  
-   대화 상자는 표시 되 고 자주 또는 많은 사용자가 사용 하는 일반적인 환경 (예를 들어는 **새 프로젝트** 대화 합니다.  
  
-   주요 제품 브랜드 요소를 포함 하는 대화 상자 (예를 들어는 **계정 설정** 대화).  
  
-   테마가 지정 된 다른 대화 상자를 포함 하 여 더 큰 흐름의 필수 단계는 대화 상자가 나타납니다 (예를 들어는 **연결 된 서비스 추가** 대화).  
  
-   대화 상자에는 승격 하거나 차별화 제품 버전에서 전략상 중요 한 역할을 담당 하는 환경의 중요 한 부분입니다.  
  
 테마가 지정 된 대화를 만들 때 적절 한 환경 색을 사용 하 고 올바른 레이아웃 및 상호 작용 패턴을 따릅니다. (참조 [Visual Studio에 대 한 레이아웃](../../extensibility/ux-guidelines/layout-for-visual-studio.md))  
  
### <a name="dialog-design"></a>대화 상자 디자인  
 다음 요소를 고려해 설정 하는 잘 디자인 된 대화 상자:  
  
-   지원 되 고 사용자 작업  
  
-   텍스트 스타일 대화 상자, 언어 및 용어  
  
-   컨트롤 선택 및 UI 규칙  
  
-   시각적 레이아웃 사양 및 컨트롤 맞춤  
  
-   키보드 액세스  
  
#### <a name="content-organization"></a>내용 구성  
 이러한 기본 유형의 대화 간의 차이점을 고려해 야 합니다.  
  
-   [간단한 대화 상자](../../extensibility/ux-guidelines/application-patterns-for-visual-studio.md#BKMK_SimpleDialogs) 단일 모달 창에서 컨트롤을 제공 합니다. 프레젠테이션 필드 선택 또는 아이콘 표시줄을 포함 하 여 복잡 한 컨트롤 패턴의 변형 포함 될 수 있습니다.  
  
-   [대화 상자를 계층화 된](../../extensibility/ux-guidelines/application-patterns-for-visual-studio.md#BKMK_LayeredDialogs) 는 단일 UI 컨트롤의 여러 그룹을 구성 하는 경우의 실제 화면 공간을 최대한 활용 하는 데 사용 됩니다. 대화 상자의 그룹은 "계층화" 탭 컨트롤, 탐색 목록 컨트롤 또는 단추를 통해 사용자 지정된 된 순간에 보려면 그룹화를 선택할 수 있도록 합니다.  
  
-   [마법사](../../extensibility/ux-guidelines/application-patterns-for-visual-studio.md#BKMK_Wizards) 논리 일련의 작업의 완료 될 때까지 단계를 통해 사용자를 전달 하는 데 유용 합니다. 여러 선택 항목은 다른 워크플로 ("분기") 이전 패널에서 선택에 종속을 때때로 소개 하는 순차 패널에 제공 됩니다.  
  
####  <a name="a-namebkmksimpledialogsa-simple-dialogs"></a><a name="BKMK_SimpleDialogs"></a> 간단한 대화 상자  
 간단한 대화는 하나의 모달 창에서 컨트롤의 표시 합니다. 이 프레젠테이션에서 다양 한 필드 선택과 같은 복잡 한 컨트롤 패턴을 포함할 수 있습니다. 간단한 대화 상자에 대 한 복합 컨트롤을 그룹화 하는 데 필요한 특정 레이아웃 뿐만 아니라 표준 일반 레이아웃을 따릅니다.  
  
 ![Visual Studio의 간단한 대화 상자](../../extensibility/ux-guidelines/media/0704-01_createstrongnamekey.png "0704-01_CreateStrongNameKey")  
  
 **만들 강력한 이름 키는 Visual Studio에서 간단한 대화의 예입니다.**  
  
####  <a name="a-namebkmklayereddialogsa-layered-dialogs"></a><a name="BKMK_LayeredDialogs"></a> 계층화 된 대화 상자  
 계층화 된 대화 상자 탭, 대시보드 및 포함 된 트리를 포함 합니다. UI의 한 부분에서 제공 하는 컨트롤의 여러 그룹에 있는 경우 부동산을 최대화 하기 위해 사용 됩니다. 사용자는 한 번에 보려면 그룹화를 선택할 수 있도록 그룹화 계층화 됩니다.  
  
 가장 간단한 경우, 그룹 간 전환을 메커니즘 탭 컨트롤입니다. 수 있는 여러 가지 대안 중 있습니다. Prioritizing 및 가장 적절 한 스타일을 선택 하는 방법에 대 한 계층 처리를 참조 하십시오.  
  
  **도구 > 옵션** 대화 상자는 포함된 하는 트리를 사용 하 여 계층화 된 대화의 예:  
  
 ![Visual Studio의 계층화된 대화 상자](../../extensibility/ux-guidelines/media/0704-02_toolsoptions.png "0704-02_ToolsOptions")  
  
 **도구 > 옵션은 Visual Studio의 계층화 된 대화의 예입니다.**  
  
####  <a name="a-namebkmkwizardsa-wizards"></a><a name="BKMK_Wizards"></a> 마법사  
 마법사는 논리는 일련의 단계를 통해 사용자는 작업의 완료에 모으는 데 유용 합니다. 순차 패널에 제공 되는 여러 선택 항목 및 사용자는 다음 단계로 진행 하기 전에 각 단계를 계속 해야 합니다. 한 번의 충분 한 기본값을 사용할 수는 **마침** 단추를 사용할 수 있습니다.  
  
 작업에 대 한 모달 마법사를 사용 하는:  
  
-   분기, 사용자의 선택에 따라 서로 다른 경로 제공 하는 위치를 포함 합니다.  
  
-   이후 단계에서는 이전 단계에서 사용자 입력에 따라 달라 집니다 단계 간의 종속성을 포함 합니다.  
  
-   제공 된 선택 사항 및 각 단계에서 가능한 결과 설명 하는 UI를 사용 해야 함을 복잡 한 충분히  
  
-   트랜잭션, 모든 변경 내용이 커밋되기 전에 전체에 완료할 수 있도록 단계 집합이 필요 합니다.  
  
### <a name="common-conventions"></a>일반 규칙  
 최적의 디자인 및 기능이 대화 상자를 위해 대화 크기, 위치, 표준, 제어 구성 및 맞춤, UI 텍스트, 제목 표시줄, 컨트롤 단추 및 액세스 키에 이러한 규칙을 따릅니다.  
  
 레이아웃 관련 지침을 참조 하십시오. [Visual Studio에 대 한 레이아웃](../../extensibility/ux-guidelines/layout-for-visual-studio.md)합니다.  
  
#### <a name="size"></a>크기  
 최소 1024 x 768 화면 해상도, 내 대화 상자에 포함 해야 하 고 초기 대화 크기는 900 개의 700mhz 픽셀 초과할 수 없습니다. 대화 상자를 조정할 수 있습니다 있지만 필요는 없습니다.  
  
 크기 조정 가능한 대화 상자에 대 한 두 가지 권장 사항 가지가 있습니다.  
  
1.  최소 크기에서 컨트롤에 대해 최적화 되는 대화 상자에 대 한 정의 된 것을 자르지 않고 설정 하 고 적절 한 지역화 증가 맞게 조정 합니다.  
  
2.  사용자 확장 크기는 세션 간에 지속 됩니다. 예를 들어, 사용자를 150% 대화를 확장 하는 경우 대화 상자의 이후 출시 150%로 표시 됩니다.  
  
#### <a name="position"></a>위치  
 대화 상자에서 처음 시작 하면 IDE 내에서 가운데에 나타나야 합니다. 크기 조정이 불가능 대화 상자는 것이 대화의 마지막 위치를 유지 함이 필요한 이후에 시작에 집중 표시 됩니다. 크기 조정 가능한 대화 상자에 대 한 크기 이후에 시작에 유지 되어야 합니다. 크기를 조정할 수 있는 모달 대화 상자에 대 한 위치를 유지할 필요가 없습니다. IDE 내에서 가운데 맞춤으로 표시 사용자의 디스플레이 구성이 변경 될 때 예기치 않은 또는 사용할 수 없는 위치에 표시 되는 대화의 가능성을 방지 합니다. 위치가 변경 될 수 있는 모덜리스 대화 상자에 대 한 더 큰 워크플로의 일부로 대화 상자를 자주 사용 될 수 있습니다 이후에 시작에 사용자의 위치를 유지 해야 합니다.  
  
 맨 위에 있는 대화 상자 오른쪽에 연계 해야 대화 상자는 다른 대화 상자를 생성 해야 때 및 그 공급자는 새 위치로 이동 하는 사용자에 게 명백 하 게 되므로 부모 로부터 다운 합니다.  
  
#### <a name="modality"></a>형식  
 모달 되 고 완료 하거나 계속 하기 전에 대화 상자를 취소 하는 데 필요한 사용자를 의미 합니다. 모달 대화 상자 사용자 환경의 다른 부분과 상호 작용을 차단 하므로 기능의 작업 흐름이 해야으로 너무 많이 사용 가능한 합니다. 모달 작업이 필요한 경우 Visual Studio에 다양 한 공유 대화 상자에 해당 기능을 통합할 수 있습니다. 새 대화 상자를 만들어야 하는 경우 유사한 기능을 가진 기존 대화의 상호 작용 패턴을 따릅니다.  
  
 와 같이 한 번에 두 가지 작업을 수행 하려면 사용자가 할 **찾을** 및 **대체** 새 코드를 작성 하는 동안 대화 해야 모덜리스 사용자 간에 쉽게 전환할 수 있도록 합니다. Visual Studio는 이러한 종류의 편집기를 지 원하는 연결 된 작업에 대 한 도구 창을 일반적으로 사용합니다.  
  
#### <a name="control-configuration"></a>제어 구성  
 Visual Studio에서 같은 작업을 수행 하는 기존 컨트롤 구성으로 동일 합니다.  
  
#### <a name="title-bars"></a>제목 표시줄  
  
-   제목 표시줄의 텍스트를 실행 하는 명령 이름을 반영 해야 합니다.  
  
-   대화 상자 제목 표시줄에 아이콘이 사용 해야 합니다. 시스템 필요 하나의 경우에서 Visual Studio 로고를 사용 합니다.  
  
-   대화가 필요 하지 않은 최소화 또는 최대화 단추가 있습니다.  
  
-   제목 표시줄에 도움말 단추가 있는 더 이상 사용 되지 않습니다. 이러한 새 대화 상자에 추가 하지 마십시오. 없는 경우 작업에 개념적으로 관련이 있는 도움말 항목 시작 해야 합니다.  
  
 ![Visual Studio의 제목 표시줄 사양](../../extensibility/ux-guidelines/media/0704-03_titlebarspecs.png "0704-03_TitleBarSpecs")  
  
 **Visual Studio 대화 상자에서 제목 표시줄에 대 한 지침 사양입니다.**  
  
#### <a name="control-buttons"></a>컨트롤 단추  
 일반적으로 **확인**/**취소**/**도움말** 대화 상자의 오른쪽 아래 모퉁이의 단추를 가로로 정렬 해야 합니다. 대화 상자에 컨트롤 단추와 시각적 혼동을 발생 하 게 되는 대화 상자 맨 아래에 있는 다른 여러 단추를 포함 하는 경우 대체 세로 스택을 허용 됩니다.  
  
 ![Visual Studio의 컨트롤 단추 구성](../../extensibility/ux-guidelines/media/0704-04_controlbuttonconfig.png "0704-04_ControlButtonConfig")  
  
 **Visual Studio 대화 상자에서 컨트롤 단추에 대 한 허용 가능한 구성**  
  
 대화 상자에는 기본 컨트롤 단추를 포함 해야 합니다. 기본적으로 사용 하 여 최상의 명령을 확인 하려면 (우선 순위 대로 나열) 다음 옵션 중에서 선택 합니다.  
  
-   기본적으로 가장 안전 하 고 가장 안전한 명령을 선택 합니다. 즉, 데이터 손실을 방지 하 고 의도 하지 않은 시스템 액세스를 방지 가능성이 가장 높은 명령을 선택 합니다.  
  
-   데이터가 손실 되 고 보안에 요소가 없으면 편의에 따라 기본 명령을 선택 합니다. 기본적으로 가능성이 가장 높은 명령을 포함 하 여 대화 상자에 자주 또는 반복적인 작업을 지 원하는 경우 사용자의 워크플로 향상 됩니다.  
  
 기본 명령에 대 한 영구적으로 삭제 작업을 선택 하지 마십시오. 이 명령을 있는 경우 안전 하 게 명령을 대신 기본적으로 선택 합니다.

 에 대 한 액세스 키를 사용 하지 않는 **확인**/**취소**/**도움말** 단추입니다. 이러한 단추는 기본적으로 바로 가기 키에 매핑됩니다.

|-----------------|-----------------------|  
|확인을 클릭합니다|입력|

|도움말|F1|  
  
#### <a name="imagery"></a>이미지  
 대화 상자에서 필요한 경우에 이미지를 사용 합니다. 공간을 사용 하 여 단순히 대화 상자에서 큰 아이콘을 사용 하지 마십시오. 경고 아이콘 또는 상태 애니메이션 등 사용자에 게 메시지 전달의 중요 한 부분이 있는 경우에 이미지를 사용 합니다.  
  
###  <a name="a-namebkmkprioritizingandlayeringa-prioritizing-and-layering"></a><a name="BKMK_PrioritizingAndLayering"></a> 우선 순위 지정 및 레이어로 표시  
  
#### <a name="prioritizing-your-ui"></a>UI를 우선 순위 지정  
 특정 UI 요소를 가져오고 고급 동작 및 대화 상자에 옵션 (모호한 명령 포함)를 배치 해야 할 수도 있습니다. 에 대 한 공간을 마련 하 고 있도록 하 여 표시 되는 텍스트 레이블을 UI에는 기본적으로 대화 상자를 표시 하는 경우 자주 사용 되는 기능 forefront을 부여 합니다.  
  
#### <a name="layering-your-ui"></a>UI를 레이어로 표시  
 대화 상자에 필요한 이지만 간단한 대화에 사용할 수 있는 기능 외의 작업도 수행할 사용자에 게 표시 하려는 관련된 기능을 결정 하는 경우 UI 계층 해야 합니다. Visual Studio를 사용 하는 가장 일반적인 쌓기 메서드는 탭 및 복도 또는 대시보드입니다. 경우에 따라 확장 및 축소할 수 있는 영역 적절할 수 있습니다. 적응 UI 일반적으로 Visual Studio에서 권장 되지 않습니다.  
  
 장점과 단점을 레이어로 유사한 탭 컨트롤을 통해 UI를 표시 하는 다양 한 방법 있습니다. 상황에 맞는 쌓기 기법을 선택 했는지 확인 하려면 아래 목록을 검토 합니다.

##  <a name="a-namebkmkprojectsa-projects"></a><a name="BKMK_Projects"></a> 프로젝트  
  
### <a name="projects-in-the-solution-explorer"></a>솔루션 탐색기의 프로젝트  
 대부분의 프로젝트와 참조 기반, 디렉터리 기반 또는 혼합 분류 됩니다. 솔루션 탐색기에서 프로젝트의 세 가지 유형은 모두 동시에 지원 됩니다. 이 창 내에서 프로젝트를 작업할 때 사용자 경험의 루트가 이루어집니다. 다른 프로젝트 노드 참조, 디렉터리 또는 혼합 형식 프로젝트를 프로젝트에 특정 사용자의 패턴에 수렴 하기 전에 시작 지점으로 적용 해야 하는 일반적인 상호 작용 패턴이 있습니다.  
  
 프로젝트 항상 수행 해야합니다.  
  
-   프로젝트 내용을 구성 하기 위해 프로젝트 폴더를 추가 하는 기능을 지원 합니다.  
  
-   프로젝트 지 속성에 대 한 일관 된 모델을 유지 관리  
  
 프로젝트에 대 한 일관 된 상호 작용 모델 유지 관리도 해야 합니다.  
  
-   프로젝트 항목 제거  
  
-   문서 저장  
  
-   프로젝트 속성 편집  
  
-   대체 보기에서 프로젝트를 편집합니다.  
  
-   끌어서 놓기 작업  
  
### <a name="drag-and-drop-interaction-model"></a>끌어서 놓기 상호 작용 모델  
 프로젝트에 자체 참조 기반으로 (저장소에 대 한 프로젝트 항목에 대 한 참조만 유지할 수) 항목을 일반적으로 분류 디렉터리 기반 (수만 프로젝트 항목을 물리적으로 유지 하기 위해 저장 된 프로젝트의 계층 구조 내에서), 또는 혼합 (참조 또는 실제 항목을 유지 하기 위해 수). IDE 내에서 동시에 모든 세 가지 유형의 프로젝트를 수용는 **솔루션 탐색기**합니다.  
  
 끌어서 놓기 관점에서 보면 다음과 같은 특징이 내에서 프로젝트의 각 형식에 적용 해야는 **솔루션 탐색기**:  
  
-   **프로젝트 기반 참조:** 요점은 프로젝트 저장소의 항목에 대 한 참조를 주위 끌어가 것입니다. 참조 기반 프로젝트를 소스 역할을 하는 이동 작업에 대 한 때 프로젝트에서 항목에 대 한 참조만 제거 해야 것입니다. 항목이 하드 드라이브에서 실제로 삭제 해야 합니다. 이동 (또는 복사) 작업에 대 한 대상으로 참조 기반 프로젝트 역할을 하는 경우 항목의 개인 복사본을 만들지 않고 원래 소스 항목에 대 한 참조를 추가 해야 합니다.  
  
-   **프로젝트 디렉터리 기반:** 는 끌어서 놓기 관점에서 프로젝트를 참조 하지 않고 실제 항목 주위 끌어옵니다. 디렉터리 기반 프로젝트를 소스 역할을 하는 이동 작업에 대 한,을 하드 드라이브에서 실제 항목을 삭제 하 고 프로젝트에서 제거 끝나야 합니다. 이동 (또는 복사) 작업에 대 한 대상 디렉터리 기반 프로젝트 역할을 하는 경우 대상 위치에서 소스 항목의 복사본을 확인 해야 것입니다.  
  
-   **혼합 대상 프로젝트:** 는 끌어서 놓기 관점에서 이러한 유형의 프로젝트의 동작 중 하나 (저장소의 항목에 대 한 참조) 또는 항목 자체 끌고 있는 항목의 특성에 기반 합니다. 실제 항목 및 참조에 대 한 올바른 동작은 위에 설명 되어 있습니다.  
  
 한 가지 유형의 프로젝트에 있는 경우는 **솔루션 탐색기**, 끌어서 놓기 작업을 간단 하 게 됩니다. 각 프로젝트 시스템 자체 끌어서 놓기 동작을 정의 하는 기능을 사용 하므로 특정 지침 (Windows 탐색기 끌어서 놓기 동작에 따라) 예측 가능한 사용자 환경을 유지 하기 위해 따라야 합니다.  
  
-   는 수정 되지 않은 끌기 작업의 **솔루션 탐색기** 이동 작업을 유발 (때 Ctrl 또는 Shift 키를 모두는 누른 상태).  
  
-   이동 작업이 shift 끌기 작업 수 없게 해야 합니다.  
  
-   Ctrl + 드래그 작업 복사 작업을 유발 합니다.  
  
-   프로젝트 기반 참조 및 혼합 된 시스템은 소스 항목을 링크 (또는 참조)를 추가 하는 개념을 지원 합니다. 이러한 프로젝트는 끌어서 놓기 작업의 대상을가 하는 경우 (때 **Ctrl + Shift** 를 누르고), 프로젝트에 추가 되는 항목에 대 한 참조를 그러면  
  
 일부 끌어서 놓기 작업은 참조 기반, 디렉터리 기반 및 혼합 프로젝트의 조합을 통해 합리적인입니다. 특히,은 소스 디렉터리 기반 프로젝트를 이동 하는 완료 되 면 소스 항목을 삭제 해야 합니다 때문에 소스 디렉터리 기반 프로젝트와 프로젝트 기반 참조 대상 간에 이동 작업을 허용 하도록 노테이션 문제가 있습니다. 대상 참조 기반 프로젝트 다음 결국 삭제 된 항목에 대 한 참조입니다.  
  
 대상 기반 참조 프로젝트는 소스 항목의 독립 복사본을 사용 하지 않아야 하기 때문에 이러한 유형의 프로젝트 간에 복사 작업을 허용 하도록 노테이션 오해의 이기도 합니다. 마찬가지로, Ctrl + Shift를 대상 디렉터리 기반 프로젝트를 끌어 허용 되지 디렉터리 기반 프로젝트에 대 한 참조를 유지할 수 없기 때문에 있습니다. 끌어서 놓기 작업이 지원 되지 않습니다의 경우에서 IDE의 삭제를 허용 하지 하 고 (포인터 테이블 아래에 표시) 놓기 없음 커서 사용자에 게 표시 해야 합니다.  
  
 끌어서 놓기 동작을 제대로 구현 하려면 끌기 소스 프로젝트의 특성을 통신 해야 (예를 들어 입니까 참조 또는 디렉터리 기반?) 대상 프로젝트에 있습니다. 이 정보는 원본에서 제공 하는 클립보드 형식으로 표시 됩니다. 끌어서 (또는 클립보드로 복사 작업)의 원본으로 프로젝트를 제공 해야 하거나 **CF_VSREFPROJECTITEM**S 또는 **CF_VSSTGPROJECTITEMS** 각각 인지 여부에 따라 프로젝트 기반 참조 디렉터리 기반 합니다. 사용 되는 Windows에 동일한 데이터 콘텐츠를 포함 하는 이러한 형식을 모두 **CF_HDROP** 파일 이름 대신 문자열의 목록이 이중-점을 제외 하 고 서식을 지정**NULL** 목록이 종료 **Projref** 문자열 (에서 반환 된 **IVsSolution::GetProjrefOfItem** 또는 **:: GetProjrefOfProject** 필요에 따라).  
  
 Drop (또는 클립보드 붙여넣기 작업)의 대상으로 프로젝트를 둘 다 허용 **CF_VSREFPROJECTITEMS** 및 **CF_VSSTGPROJECTITEMS**, 경우에 끌어서 놓기 작업의 처리는 정확한 대상 프로젝트 및 원본 프로젝트의 성격에 따라 달라 집니다. 제공 여부 여의 특성을 선언 하는 소스 프로젝트 **CF_VSREFPROJECTITEMS** 또는 **CF_VSSTGPROJECTITEMS**합니다. 끌어 놓기 대상 고유한 특성을 이해 하 고 이동, 복사, 여부를 결정을 내릴 수 있는 충분 한 정보에 있으므로 또는 링크를 수행 해야 합니다. 또한 사용자는 Ctrl, Shift, 또는 둘 다 Ctrl 및 Shift 키를 눌러는 끌어서 놓기 작업이 수행 될 수정 합니다. 반드시 나타내려면 제대로 어떤 작업이 수행 됩니다. 미리에 놓기 대상에 대 한 해당 **DragEnter** 및 **끌기** 메서드.  **솔루션 탐색기** 소스 프로젝트와 대상 프로젝트 인지 동일한 프로젝트를 자동으로 감지 합니다.  
  
 Visual Studio의 인스턴스를 통해 (예: 다른 devenv.exe 인스턴스 하나)에서 프로젝트 항목을 끌어는 특별히 없습니다.  **솔루션 탐색기** 도 직접이 사용 하지 않습니다.  
  
 항상 사용자 대상 위치로 끌어을 관찰 다음 마우스 포인터의 나타나는 항목은 삭제 하기 전에 항목을 선택 하 여 끌어서 놓기 작업의 효과 확인할 수 있습니다.

-   열림 또는 닫힘 끕니다 (위 시나리오에서 언급 되지) 시간에 있는 파일 간의 동작 차이 있습니다.  
  
-   최상위 파일 폴더의 파일 보다 약간 다르게 동작 합니다.  
  
 다른 알아야 할 사항은 열려 디자이너 또는 편집기 있는 항목에 대 한 이동 작업을 처리 하는 방법입니다. 예상 되는 동작 (모든 프로젝트 형식에 적용) 다음과 같이 수행 됩니다.  
  
1.  편집기/디자이너 열기에 변경 내용을 저장 하지 않은 경우 다음 편집기/디자이너 창이 자동으로 닫아야 합니다.  
  
2.  편집기/디자이너 열기 변경 내용을 저장 하지 않는 경우 저장을 수행 하 고 다음 다음과 유사 하 게 프롬프트를 사용 하 여 창 닫기 전에 열려 있는 문서에 커밋되지 않은 변경 내용을 저장 하려면 사용자에 게 요청에 대 한 끌기 소스 대기 해야 합니다.  
  
    ```  
    ==========================================================   
         One or more open documents have unsaved changes.  
    Do you want to save uncommitted changes before proceeding?   
                      [Yes]  [No]  [Cancel]   
    ==========================================================  
    ```  
  
 이렇게 하면 사용자를 대상의 복사본을 만들어 전에 진행 중인 작업을 저장할 수 있습니다. 새 메서드 **IVsHierarchyDropDataSource2::OnBeforeDropNotify** 이 처리를 사용 하도록 설정 하려면 추가 되었습니다.  
  
 저장소에는 대상 항목의 상태를 다음 복사 합니다 (사용자가 선택 하는 경우 편집기에서 저장 되지 않은 변경 내용을 포함 하지 않고 **No**). 대상의 복사를 완료 된 후 (에서 **IVsHierarchyDropDataSource::Drop**), 소스는 이동 작업의 삭제를 완료 하 기회가 됩니다 (에서 **IVsHierarchyDropDataSource::OnDropNotify**).  
  
 저장 하지 않은 변경 내용으로 모든 편집기 열기 두어야 합니다. 저장 하지 않은 변경 내용으로 이러한 문서에 대 한 즉 이동 작업의 복사 단계가 수행 됩니다 있지만 delete 부분 중단 됩니다. 사용자가 여러 선택 영역 시나리오에서 **No**, 문서를 저장 하지 않은 변경 종료 되거나 제거 되지 않아야 당장은 이러한 변경 내용을 저장 하지 않고 종료 하 고 제거 합니다.
