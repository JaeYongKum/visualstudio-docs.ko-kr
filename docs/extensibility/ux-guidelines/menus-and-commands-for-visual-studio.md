# 메뉴와 Visual Studio 명령
[!INCLUDE[vs2017banner](../../code-quality/includes/vs2017banner.md)]

## 명령 사용법  
  
### 개요  
 여러 개의 별도 제품을 구성 하는 제품군 인 Microsoft Office와는 달리 Visual Studio 글로벌 Visual Studio IDE에 자신의 명령 집합을 기여 각각 여러 개의 제품을 포함 합니다. IDE는 컨텍스트를 기반으로 사용자에 게 사용할 수 있는 기능을 필터링 하 여 명령의 수천의 복잡성을 관리 합니다.  
  
 관련이 없는 기능 – – 편집 창 코드를 디자인 창에서 전환 하는 등 사용자의 컨텍스트를 변경 하는 경우 새 컨텍스트가 사라집니다. 동시에, 속성 및 도구 상자 옵션 등의 관련된 동적 정보와 함께 새로운 기능 화면입니다. 사용자는 사용할 수 있는 명령 집합의 교체를 알 수 없습니다. 사용자가 방해가 또는 나타나거나 없어 졌음을 알았을 명령을 사용 하 여 혼란 스 러가 조정 해야 할 UI 디자인 합니다. 사용자의 현재 컨텍스트는 IDE 제목 표시줄, 속성 창 또는 속성 페이지 대화 상자와 같은 하나 이상의 방법으로 항상 표시 합니다.  
  
 명령 모음 UI에 유연성을 허용합니다. 유일한 명령 구조 Visual Studio 환경에는 주 메뉴 및 모두 사용자 지정 하 고 수 있는 숨겨진도 주 명령 모음에 고유 합니다. 다른 명령 모음에 나타나고 응용 프로그램의 상태에 따라 사라지게 됩니다. 도구 창과 문서 편집기의 창 가장자리에 포함 된 도구 모음을 포함할 수도 있습니다.  
  
#### 기본 지침  
  
##### 기존 공유 명령, 명령 그룹 및 가능한 메뉴를 사용 합니다.  
 명령을 컨텍스트를 기준으로 일반적으로 표시, 이후 명령 구조 컨텍스트에서 변경 내용을 간의 상대적으로 안정 된 상태로 유지 됩니다 기존 공유 메뉴 및 명령 그룹을 사용 하면 됩니다. 공유 명령 다시 사용 관련된 공유 명령 가까운 새로운 명령이 배치 IDE 복잡성 감소 및 보다 편리한 사용자 환경을 만듭니다. 새 명령을 정의 하는 경우 기존 공유 명령 그룹에 배치 하려고 합니다. 새 그룹을 정의 하는 경우 앞에 배치 관련된 명령 그룹에 가까운 기존 공유 메뉴에서 새 최상위 메뉴를 만듭니다.  
  
##### 모든 명령에 대 한 아이콘을 만들지 마십시오.  
 명령 아이콘을 만들기 전에 신중 하 게 생각 합니다. 명령에 대 한 아이콘을 만들어야 하는:  
  
-   기본 도구 모음에 표시 됩니다.  
  
-   도구 모음을 통해 사용자가 추가 될 가능성이 **각 사용자 지정** 대화 합니다.  
  
-   다른 Microsoft 제품의 동일한 작업과 연결 된 아이콘이 있습니다.  
  
##### 바로 가기 키를 추가 제한  
 대부분의 사용자는 모든 사용 가능한 바로 가기에 비해 매우 작은 사용 합니다. 확실 하지 않은 연결 하지 기능 바로 가기 키입니다. 새 바로 가기를 추가 하기 전에 팀 경험 하는 사용자와 작업.  
  
##### 명령은 기본 메뉴 배치를 제공 합니다.  
 명령을 다른 사용자가 사용자 지정 된 수와 그에 따라를 디자인할 주의 확인 하십시오. 숨겨진된 명령으로 생각해가 있습니다. 모든 Visual Studio 명령에 표시 된 **도구 \> 사용자 지정** 대화 상자에서 자동 완성, 명령 창은 **도구 \> 옵션 \> 키보드** 대화 상자 및 개발 도구 환경 \(DTE\). 사용자가 쉽게 찾을 수 있도록 이름과.ctc 파일에 대 한 도구 설명 명령을 지정 해야 합니다.  
  
##### 공유 명령에 포함된 된 도구 모음을 복제 하지 마십시오.  
 사용자의 포커스의 영역에 인접에서 명령을 배치 하는 것이 유용 합니다. 이 작업을 수행 하는 방법을 편집기 도구 창이 나 문서 맨 위에 있는 포함된 된 도구 모음을 만드는 것입니다. 도구 모음에 명령 창 내에서 콘텐츠 영역에 구체적 이어야 합니다. 이러한 도구 모음에서 공유 명령을 복제 하지 마십시오. 예를 들어 포함된 된 도구 모음 내에서 "저장" 아이콘을 두지 마십시오.  
  
### 콘텐츠 및 명령 표시 유형  
 다음 범위에 있는 명령: **환경**, **계층**, 및 **문서**합니다. 명령 배치 했으므로 신뢰 하기 위해 각 범위를 알고 있습니다.  
  
 명령에 **환경** 범위는 기본 컨텍스트를 설정 하 고 여러 컨텍스트 사이 공유 됩니다. 표시 유형 또는 문서 및 도구 창을 배열 변경 됩니다. 범위는 환경에서 명령 중 **새 프로젝트**, **서버에 연결**, **프로세스 연결**, **잘라내기**, **복사**, **붙여넣기**, **찾을**, **옵션**, **사용자 지정**, **새 창**, 및 **도움말 보기**합니다.  
  
 명령에 **계층** 비롯 하 여 Visual Studio의 계층 구조를 관리 하는 범위 **프로젝트**, **팀**, 및 **데이터**합니다. 예를 들어 프로젝트의 하위 컨텍스트로 – 관계 **디버그**, **빌드**, **테스트**, **아키텍처**, 또는 **분석**합니다. 범위는 계층 구조에서 명령 중 **새 항목 추가**, **새 쿼리**, **프로젝트 설정**, **새 데이터 소스 추가**, **성능 마법사 시작**, 및 **새 다이어그램**합니다.  
  
 명령에 **문서** 범위 작업 내용에 같은 코드, 디자인 또는 작업 항목 쿼리 \(WIQ\) 문서입니다. 또한 도구 창의 보기에 대해 작동 하거나 되며, 그렇지 않으면 해당 도구 창에 특정 합니다. 문서 범위 명령을 사용 하는 등의 계층 구조 별로 하는 파일 개체에도 작동할 **프로젝트에서 제거**합니다. 범위는 문서에서 명령 중 **리팩터링 \> 이름 바꾸기**, **작업 항목 복사본 만들기**, **모두 확장**, **모두 축소**, 및 **사용자 작업 만들기**합니다.  
  
### 명령 배치 결정  
 명령을 만들려면 결정 하면, 적절 한 위치 및 바로 가기 키를 만들지 여부를 확인 해야 합니다. 명령 배치 위치를 설정 하려면이 의사 결정 경로를 따릅니다.

### 메뉴에 명령 배치  
  
#### 주 메뉴 모음  
 주 메뉴 모음에는 UI에 영향을 주는 상황에 맞는 메뉴 패키지의 명령에 대 한 표준 위치 여야 합니다. 주 메뉴 모음 환경을 사용 하 여 볼 수 있는 명령을 제어 점에서 다른 명령 구조와 다릅니다. 다른 모든 명령 모음 명령을 컨텍스트를 벗어나는 해제할 메뉴 또는 도구 모음에 저장 되기 여부입니다.  
  
 환경에는 전체 IDE 및 다중 작업 도메인에서 공통 된 주 메뉴 모음에 기본 제공 명령 집합을 정의 합니다. 이러한 명령은 항상 표시 됩니다에 관계 없이는 VSPackages 환경에 로드 됩니다. 각 제품 위치와 해당 명령 명령 집합 Vspackage 명령이 집합을 확장할 수 있습니다 이지만 각 팀의 책임입니다.  
  
 Visual Studio 주 메뉴의 구조 메뉴는 다음과 같은 범주로 구분할 수 있습니다.  
  
##### 핵심 메뉴  
  
-   파일  
  
-   편집  
  
-   보기  
  
-   도구  
  
-   창  
  
-   도움말  
  
##### 프로젝트 관련 메뉴가  
  
-   프로젝트  
  
-   빌드  
  
-   디버그  
  
##### 상황에 맞는 메뉴  
  
-   팀  
  
-   데이터  
  
-   테스트  
  
-   아키텍처  
  
-   분석  
  
##### 문서 관련 메뉴  
  
-   서식  
  
-   표  
  
##### 주 메뉴를 설계할 때 다음이 규칙을 따라야 합니다.  
  
-   특정 컨텍스트에서 최상위 25 개 항목을 초과 하지 않는  
  
-   메뉴 높이 600 픽셀을 초과할 수 없습니다.  
  
-   Ultimate SKU와 일반 프로필에서와 같은 여러 컨텍스트에서 주 메뉴를 평가 합니다.  
  
-   플라이 아웃 메뉴를 사용할 수 있습니다.  
  
-   플라이 아웃 메뉴는 최소한 세 개 항목 및 7 개 이하의 포함 되어야 합니다.  
  
-   플라이 아웃 메뉴 보내야 하는 하나의 수준만 심층 – 일부 Visual Studio 메뉴 항목 연계 메뉴가 있지만이 패턴은 권장 되지 않지만.  
  
-   최대 6 개의 구분 기호를 사용 합니다. 그룹화는 다음 그림을 따라야 합니다.  
  
     ![Guidelines for main menu grouping](../../extensibility/ux-guidelines/media/0501-b_mainmenus.png "0501\-b\_MainMenus")  
  
-   그림에서 각 그룹화 할 필요는 없습니다, 하는 동안 추가 그룹화 추가 제한 됩니다.  
  
-   각 그룹화 2 ~ 7 메뉴 항목에 있어야 합니다.  
  
#### 주 메뉴 순서 지정  
 새 최상위 항목을 추가 하기 전에 기존 최상위 메뉴에 명령이 배치 하는 것이 좋습니다. 새 최상위 메뉴를 추가할 때 올바른 위치에 배치 해야 합니다. 메뉴는 프로젝트, 컨텍스트 또는 문서와 관련이 있는지 여부를 결정 합니다. 최상위 메뉴의 이름을 간결 하 게 유지 하 고 하나의 단어를 사용 합니다.  
  
 핵심 메뉴 북엔드 나머지 명령 해야 합니다. 파일, 편집 및 보기에 항상 왼쪽 및 도구, 창 이어야 하며 도움말 오른쪽에 항상 있어야 합니다.  
  
#### 상황에 맞는 메뉴  
 상황에 맞는 메뉴 내에서 너무 많은 기능을 배치 하면 배우기 어렵습니다 인터페이스에서 발생 합니다. 모든 주요 기능 주 메뉴 모음을 통해 제공 해야 합니다. 명령 배치 명령이 중복을 방지 하려면 기존 명령을 사용 하 여 조정 해야 합니다. 상황에 맞는 메뉴에 대 한 셸 솔루션, 프로젝트 노드 또는 프로젝트 항목에 대 한 상황에 맞는 메뉴 인지에 따라 포함 되어야 하는 표준 메뉴 그룹을 정의 합니다.  
  
 상황에 맞는 메뉴를 디자인할 때에 또한 주 메뉴의 경우와 동일한 규칙을 따릅니다.  
  
-   25 최상위 메뉴 항목을 초과 하지 않습니다.  
  
-   플라이 아웃 메뉴는 사용할 수 있지만 해야 한 수준 깊이 초과 하지 – 연계 플라이 아웃을 사용 하지 마십시오.  
  
-   최대 6 개의 구분 기호를 사용 합니다.  
  
### 도구 모음에 명령 배치  
  
#### 일반 도구 모음  
 를 디자인 하 고 도구 모음을 정렬 하는 경우에 이러한 표준을 따르십시오.  
  
-   단추를 별로 여러 개 동사를 사용 하지 마십시오. 한 단추 \= 한 동작입니다.  
  
-   레이블로 보강 되어야 하는 데 필요한 경우에 아이콘 함께 텍스트를 사용 합니다.  
  
-   한 세션에서 여러 번 전환 됩니다 속성에 대 한 단독으로 콤보 상자를 사용 합니다. 그렇지 않은 경우 다른 곳에서 속성을 노출 합니다.  
  
-   콤보 상자의 너비 상자 \+ 30% 내에서 가장 긴 항목의 너비와 같아야 합니다. 예를 들어, 가장 긴 항목은 200 픽셀, 다시 콤보 상자 야 260 픽셀 너비입니다.  
  
-   구분 기호 사용을 제한 합니다. 옆의 드롭다운에는 구분 기호 사용은 안티패턴 이므로 자체에 있는 드롭다운의 모양을 시각적 구분 기호 역할도 합니다.  
  
-   아이콘 그룹에서 3 6 개의 아이콘을 포함 해야 합니다.  
  
-   한정자 여러 유용한 명령이 결과 경우 마지막 설정을 저장 하는 분할 단추를 사용 합니다.  
  
     ![Split buttons in Visual Studio](../../extensibility/ux-guidelines/media/0501-c_splitbuttons.png "0501\-c\_SplitButtons")  
  
     **분할 단추의 예입니다. 왼쪽에 있는 6 개의 명령을 한 번의 단추에 대신 맞출 수 있습니다.**  
  
#### 제품 관련 도구 모음  
 각 제품에 기본 도구 모음 포함 하는 데 자주 사용 하 고 중요 한 명령 및 각 제품의 기본 도구 모음 표시 되는 제품이 설치 된 후 Visual Studio 시작 처음으로 제공할 수 있습니다.  
  
 제품 공유 명령 그룹 및 IDE에서 제공 하는 메뉴에도 활용 해야 합니다. 각 공유 명령 그룹 관련된 명령을 사용자에 대 한 의미 있는 방식으로 구성 하는 공유 메뉴에 배치 됩니다. 이 공유 명령 구조를 활용 하 여 복잡성을 줄이기 위해을 고려해 야 합니다.  
  
#### 전역 도구 모음  
 전역 도구 모음은 하나의 행을 즉시에 맞게 필요 합니다. 새 전역 도구 모음을 만들 때 해당 도구 모음 형식에 대 한 지침을 따릅니다.  
  
 **일반 도구 모음 지침:**  
  
-   각 도구 모음 공용 컨트롤 \(위치 조정 막대, 오버플로\)에 24 픽셀에 있습니다.  
  
-   각 도구 모음 단추는 여백을 포함 하는 넓은 22 픽셀입니다. 분할 단추 아이콘을 만드는 다른 11 픽셀 너비의 추가 합니다.  
  
-   도구 모음에서 명령의 중복이 허용 됩니다.  
  
 **문서 관련 도구 모음** 특정 파일 형식을 상태인 경우에 표시 되 고 다른 파일 형식 활성화 되 면 사라집니다.  
  
-   문서 관련 도구 모음에는 12 개 이상의 단추를 가질 수 없습니다.  
  
-   도구 모음에서의 전체 너비는 300 픽셀을 초과할 수 없습니다.  
  
-   각 파일 형식 중 하나 포함 된 도구 모음 또는 문서 관련 전역 도구 모음, 하나만 있을 수 있습니다.  
  
 **상황에 맞는 도구 모음** 특정 컨텍스트를 표시 합니다. 설정 하 고 오랜 시간 동안 활성 상태로 유지 하는 경향이 있습니다.  
  
-   모든 상황에 맞는 도구 모음에 대 한 단추 제한을 18입니다.  
  
-   컨텍스트 활성화 되어 있을 때 대부분의 사용자가 없습니다이 도구 모음이 명령의 일관 되 게 사용 하 고, 하는 경우 다음 연결 하지 않음이 도구 모음 컨텍스트를 사용 합니다.  
  
-   도구 모음에서 컨텍스트를 종료할 때 사라집니다 있는지 확인 합니다. 이러한 도구 모음 중 시작 시에 나타나야 합니다.  
  
 **컨텍스트가 없는 도구 모음** 자동으로 표시 되지 않습니다. 만 사용자 활성화 되는 시기에 표시 됩니다. 200 픽셀 아래 최대 너비를 유지 합니다.  
  
### 일반 조직과 셸 정의 된 그룹  
 기존 공유 명령, 명령 그룹 및 메뉴를 사용 합니다. 새 명령을 정의 하는 경우 기존 공유 명령 그룹에 배치 하려고 합니다. 새 그룹을 정의 하는 경우 새 최상위 메뉴를 만들기 전에 관련된 명령 그룹에 가까운 기존 공유 메뉴에 배치 하려고 합니다. 이렇게 하면 IDE에서 일관 된 명령 배치를 유지 하면서 명령 복잡성을 줄어듭니다.  
  
 공유 **형식** 일반적으로 디자이너 스타일 문서 창의 컨텍스트에서 표시 메뉴에서 다음 그림에 나와 있습니다.

 명령은 일반적으로 기준으로 표시 됩니다 컨텍스트, 어느 시점에서 사용자에 게 표시 하는 명령 수를 줄일 수 있도록 합니다. 그러나 또한 기존 공유 메뉴를 다시 사용 해야 하 고 컨텍스트에서 변경 내용을 간에 안정적인 명령 구조를 상대적으로 유지 하기 위해 명령 그룹.

## 액세스 및 바로 가기 키  
  
### 개요  
 키보드 키 할당 두 종류가 있습니다.  
  
-   **선택키가** \(가속기\) UI 대화 상자에서 메뉴를 통해 각 레이블에 및 명령 실행에 대 한 키보드 액세스를 허용 합니다. 액세스 키는 주로 편리한 액세스를 위해, 모든 메뉴 및 대부분의 대화 상자 컨트롤에 할당 된, 외우 수, 현재 창만 영향을 줄 것은 아니기 있으며 지역화 됩니다.  
  
-   **바로 가기 키** 컨트롤 \(Ctrl\) 및 \(Fn\) 함수 키 시퀀스를 가장 자주 사용 합니다. 사용자가 사용자 더 고급 사용자와 지원 생산성 향상에서 합니다. 가장 자주 사용 되는 명령에만 할당 된 하며 주 메뉴를 무시 하는 동안 빠른 액세스를 허용 합니다. 외우 될 바로 가기 키를 사용 하려고 하 고 하는 이유 할당 해야 프로필 구성표와 일치 합니다. 바로 가기 키 구성표 프로필 마다 다를 수 있습니다. 사용자를 통해 바로 가기 키를 사용자 지정할 수 있습니다 **도구 \> 옵션 \> 키보드**합니다.  
  
### 선택 키 할당  
 Alt \+ 영숫자 키의 액세스 키 구성 됩니다. 예외 없이 각 메뉴 항목에 액세스 키를 할당 합니다. Windows 및 선택 키 할당에 대 한 일반적인 규칙을 따릅니다. 예를 들어, 액세스 키를 **파일 \> 새로 만들기** 은 항상 **F, Alt, N**합니다.  
  
 이러한 방화벽은 구분 하기 어려운 하강 문자 \(g, j, p, q, y\)와 함께 문자를 사용 하지 않도록 및 'i' \(대문자 또는 소문자\) 또는 소문자 'l'와 같은 단일 픽셀 너비 문자를 사용 하지 않는 마십시오.  
  
 가능 하면 중복 키를 사용 하지 마십시오. 복제를 피할 수 없는 경우 메뉴 시스템 키를 사용 하는 모든 명령이 순환 하 여 충돌을 처리 합니다. 예를 들어 "N" 액세스 키를 복제 하는 파일 메뉴 명령을 "숫자" 가상으로 **F, Alt, N** 새 파일을 만듭니다 및 **Alt, F, N, N** "Number" 명령을 수행 합니다.  
  
### 바로 가기 키 할당  
 모든 명령에 필요 하지 않은 시스템 \(및 사용자 메모리\)를 지나치게 사용 하는 경우 세금 있기 때문에 새로운 바로 가기 키에 할당 되지 않도록 합니다. Ceip \(사용자 환경 개선 프로그램\)에서 데이터는 Visual Studio 사용자에 게 통합된 된 바로 가기의 작은 하위 집합만 사용 함을 나타냅니다.  
  
 바로 가기 키를 정의할 때는 다음이 규칙을 따르십시오.  
  
-   **컨트롤 \(Ctrl\) 및 \(Fn\) 함수 키 시퀀스를 사용 합니다.**  
  
-   **자주 사용 되는 바로 가기 키를 유지 합니다.** 가장 인기 있는 바로 가기 키를 유지 합니다.  
  
-   **입력 하기 쉬운 편집기 바로 가기 키를 확인 합니다.** 개발자가 필요한 코드를 작성 하는 동안 가장 하는 명령에 입력 하기 쉬운 바로 가기 키를 바인딩하십시오. 예를 들어 **Edit.InvokeSmartTag** 빠른 바로 가기 키 Ctrl 같은 있어야 \+ 및 하지: Alt \+ Shift \+ F10 합니다.  
  
-   **일관 되 게 테마가 지정 된 바로 가기 위해 노력 합니다.**  
  
-   **Windows를 사용 하는 한정자 키를 확인 하는 지침을 따릅니다.** Ctrl 키 조합을 사용 하 여 문서 전체에 적용 되는 명령 등과 같이 대규모 발생 하는 명령. Shift 키 조합을 사용 하 여 확장 하거나 표준 바로 가기 키의 동작을 보완 하는 명령. Ctrl \+ Alt 조합을 사용 하지 마십시오.  
  
-   **불필요 한 바로 가기 키를 제거 합니다.** 레거시 기능을 사용 하는 경우에 액세스 키 동일한 명령에 대 한 빠른 액세스를 제공 하는 경우 \(적은 보다 10 번에서 CEIP 데이터\) 극단적인 infrequency 이하의 보통 infrequency \(100 번 보다 CEIP 데이터에서\)에 사용 되는 바로 가기를 제거 하는 것이 좋습니다. 예를 들어: Alt, H, C 도움말\/내용을 열립니다.  
  
 바로 가기 가용성을 확인 하는 간단한 방법을 않습니다. 바로 가기를 추가 하려면 다음이 단계를 따르십시오.  
  
1.  목록을 확인 하 여 [Visual Studio 2013 바로 가기를](http://visualstudioshortcuts.com/2013/) 그룹화 된 것과 유사한 명령이 되는지 확인할 수 있습니다.  
  
2.  이동 **도구 \> 옵션 \> 환경 \> 키보드** 및 바로 가기가 테스트 합니다. "다음 추가 키보드 매핑 구성표 적용 됩니다." 아래 나열 된 각 키보드 매핑 구성표를 확인 합니다. 고유 바로 가기 키를 공유 하는 것 처럼 일반, C\#, VB 및 c \+ \+ 프로필을 확인 합니다. 바로 가기는 해당 위치에 매핑되지 않은 경우에 사용할 수 있습니다.
